# Weekly Calibration Protocol (D-0047)

Исполняемый чек-лист еженедельной калибровки (ROADMAP Phase 1.5,
шаг 3). Загружается на время прогона, НЕ при boot (D-0038). До
D-0047 проверки были рассеяны по текстам решений (D-0042/44/45/46,
правила 1–10) без единого списка — калибратору пришлось бы
пере-выводить их каждый раз; этот файл — их канонический дом.
Новый механизм с обещанием «калибровка заметит X» обязан тем же
коммитом добавить сюда чек (иначе обещание пустое — F-15).

Входные данные: logs/routing-log.jsonl обоих деплоев (OS-репо и
D:\AO3_tests), cc_usage через tools/usage_report.py, git-история
обоих репо за период, диффы DECISIONS.md.

## Чек-лист (механизм → нарушение → как проверить)

1. **Правила 1/2/8 — молчаливое самоизъятие.** Main-сессии в
   cc_usage с крупным объёмом чтения/правок, у которых за тот же
   период нет ни delegated, ни dispatch_skipped. Источник:
   usage_report + журнал.
2. **Правило 3 — critic на крупных диффах.** Коммиты периода
   со >100 изменённых строк, принятые без delegated(critic) и без
   пометки «critic: skipped, <причина>» в accepted-событии.
   Источник: git log --stat + журнал.
3. **Правило 6 / D-0045 — эскалация.** Два `rejected` по одной
   задаче на одном ярусе без последующего `escalated` = нарушение.
   `escalated` без следа `rejected` = вопрос к самодекларации.
   С D-0053 считается механически: события одной задачи связаны
   `task_id`, попытки — поле `attempt`; post-D-0053 событие из
   task_id-обязательного набора без task_id = нарушение само по себе.
4. **D-0042 — самодекларация модели.** Поле model в
   delegated/escalated/accepted/rejected против фактических моделей
   исполнения в cc_usage; расхождение — само по себе событие
   калибровки.
5. **D-0039/D-0044/D-0056 — деградация.** Каждое `lead_restored`
   содержит в notes разбор окна (что ревьюлось, вердикт; пустое окно
   отмечено). `lead_degraded` без парного `lead_restored` к границе
   сессии — сверить с последним событием журнала. С D-0056 (F-21):
   фактические модели Lead-сессий периода (main-цепочки cc_usage,
   НЕ сайдчейны) против покрытия окон парами
   lead_degraded/lead_restored обоих журналов: Lead-активность на
   модели ниже Fable вне заявленного окна = молчаливая деградация —
   отказ обеих in-session точек п.4 CLAUDE.md, вкл. сессию, умершую
   деградированной (расширение чека 4 / D-0042 с воркеров на Lead;
   ретроактивные пары ищутся по пометке в notes).
6. **D-0037/D-0046/D-0058 — приёмка.** Каждый завершённый delegated
   имеет accepted или rejected. Accepted по scout-диспатчам содержит
   отметку точечной сверки (негативные утверждения — обязательно).
   Приёмка сверху (D-0058): для каждого accepted сверить ярус
   принимающей сессии (cc_usage, main chain на момент события) с
   ярусом agent — принимающий строго выше: ок; равный/ниже: в notes
   обязан быть вход яруса выше (critic) или пометка «в очередь
   полного Lead», иначе нарушение (самосертификация сессии, F-22).
   Льгота «critic: skipped» легальна только у принимающего выше
   исполнителя; смотреть вместе с чеком 5 (окна деградации).
   Env-негатив: негативные утверждения О СРЕДЕ в отчётах/вердиктах
   окна (нет сервиса/команды/файла) валидны только с позитивной
   сверкой канонической формой (командная гигиена п.6; порт AO3
   fb41872).
7. **Журнал ↔ транскрипты, обе стороны.** Сайдчейн в cc_usage без
   delegated-события = незадокументированный диспатч; delegated без
   сайдчейна = фантомное событие.
8. **Правило 10 — три вопроса.** Новые записи DECISIONS.md за период
   содержат ответы (а)/(б)/(в), причём (в)-ответ ЗАРЕГИСТРИРОВАН
   (D-0049): чек в этом протоколе тем же коммитом либо явный внешний
   детектор в тексте решения. Незарегистрированный детектор =
   механизм-пожелание, оформляется находкой. С D-0055: (б)-ответ —
   осевым ПЕРЕЧИСЛЕНИЕМ по текущей карте (проза «оси покрыты» = нет
   ответа); живость гейта — `git config core.hooksPath` равен
   .githooks в ОБОИХ репо; механизмные коммиты периода выборочно
   проверяются на осевой блок (обойти гейт можно было только
   --no-verify или строкой отказа); строки «оси: не-механизм» за
   период аудируются на честность — отказ на содержательном
   механизме = нарушение.
9. **D-0043/SIBLING_MAP — классовая полнота.** Коммиты, закрывающие
   findings, называют оси; строки «новая ось для SIBLING_MAP» из
   AO3 HANDOFF/журнала перенесены в карту.
10. **F-8 / D-0038 — рост живых файлов.** Число строк файлов
    boot-пути ОБОИХ деплоев (OS: список BOOT.md + CLAUDE.md +
    CURRENT_CONTEXT.md; AO3: CLAUDE.md + docs/HANDOFF.md — ось 1:
    автозагружаемый контекст пилота такой же платный) плюс
    docs/SIBLING_MAP.md (D-0048: карта обязана оставаться малой);
    сравнить с прошлым прогоном (числа — в notes события
    `calibrated`). Монотонный рост живого файла = нарушение D-0038
    (закрытое не архивируется).
11. **Rule #1 к самой машинерии (вопрос 10а ретроспективно).**
    Накладные расходы роутинга по данным, а не по оценкам: доля
    Lead-tier токенов, ушедшая на журнал/приёмки/сверки/спеки,
    против стоимости делегированной работы (cc_usage: main-сессия
    vs сайдчейны за период). Если оверхед механизма систематически
    сопоставим с работой, которую он маршрутизирует, — это evidence
    для упрощения механизма (решением, не молчаливым забиванием).
12. **D-0048 — живость SIBLING_MAP.** Каждый конкретный путь,
    названный в карте, существует (обе стороны осей, оба репо);
    упомянутые правила/механизмы действуют. Мёртвый путь =
    нарушение same-commit-правила карты. Рецидивы за период
    (дефект уже чинившегося класса вне указанных картой мест)
    оформлены как находки о карте, ось поправлена. Число записей
    D-XXXX в DECISIONS.md (индекс) равно числу в
    docs/DECISIONS_FULL.md (D-0051).

13. **D-0052/D-0053 — доказательная приёмка (обе стороны), типизи-
    рованные поля.** (а) Каждый accepted по builder-диспатчу несёт
    поле `witness` — фактический вывод проверочного прогона
    (команда + результат); accepted(builder) без witness =
    нарушение. (б) События `defect_found` ссылаются полем `ref` на
    task_id исходного accepted; false-accept rate по ярусам
    (defect_found / accepted за окно) считается и пишется в notes
    события `calibrated`; систематический false-accept по ярусу —
    evidence для движения статуса таблицы ВНИЗ (Update Rule 1).
    (в) Каждый `rejected` несёт `failure_class`
    (spec / capability / recon / tooling); без класса = нарушение
    (смотреть вместе с чеком 3). Поля — D-0053; на AO3 их отсутствие
    отсекает scripts/log_append.py, в OS-репо журнал пишется
    Edit-тулом и проверяется этим чеком. События до 2026-07-08
    читаются вручную (append-only, не переписаны). Счётный скрипт
    чеков 3/13 — в очереди (builder, после первого ручного прогона).
    (г) D-0054: систематический `failure_class=spec` по ярусу —
    сигнал диспатчей без DoD; проверить последние спеки Lead на
    критерии приёмки + проверочный прогон.
    (д) Целостность task_id (D-0060): дублей task_id между
    несвязанными задачами нет — в ОБОИХ журналах (OS: t-NNN;
    AO3: описательные id). Этот же скан — residual-детектор
    TOCTOU-окна кодового гарда AO3 log_append.py (гард best-effort,
    read-then-append без лока — принято Lead 2026-07-09, t-009,
    critic finding F-A). Известный дубль — t-008 (2026-07-09,
    F-23: golden-set прогон 12:42/12:55 и task-pipeline recon
    13:20/13:28) — во всех счётах чеков 3/13 идёт как ДВЕ задачи.

14. **D-0057 — golden set разведки и regression-правило правок
    агент-промптов.** (а) `git log` окна по `.claude/agents/*.md`
    обоих деплоев: каждая правка role-файла яруса, имеющего golden
    set (сейчас — scout), и каждая смена `model:` во frontmatter
    сопровождаются строкой прогона в Runs log
    PROCESS/SCOUT_GOLDEN_SET.md тем же коммитом; правка без прогона
    = нарушение. (б) Живость ключа: у >=2 вопросов набора прогнать
    verify-команды; уехавший ключ — баг eval'а, чинится ДО выводов
    о деградации scout (CORE-Bench, docs/RELATED_WORK.md «Evals»).
    (в) Рост failure_class=recon за окно (смотреть с чеком 3) без
    внепланового прогона набора -> назначить прогон. (г) Правки
    role-файлов ярусов БЕЗ набора (builder — н-п по решению, critic
    — набор в очереди) отмечаются в notes события calibrated.

## Завершение прогона

Статусы DELEGATION_TABLE.md двигаются только по этим данным
(Update Rule 1; полная стоимость с ретраями — Update Rule 4). По
завершении — событие `calibrated` в журнал OS-репо: notes содержат
период, число разобранных событий, изменения статусов таблицы,
счётчики строк из чека 10 и false-accept rate по ярусам из чека 13. Отсутствие `calibrated` дольше 7 дней
при живом routed-трафике — нарушение, видимое строкой Last
Calibration в Boot Report каждой сессии (детектор вынесен наружу
сознательно: калибровка не может быть детектором собственного
отсутствия).
