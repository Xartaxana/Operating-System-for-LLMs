# Экзамен экономии деплоя (A/B/C, 2026-07-13)

Контролируемый замер экономии САМОЙ СИСТЕМЫ: то, что savings_report
меряет наблюдательно (факт против контрфактных цен, без контрольной
группы), экзамен меряет сравнением трёх плеч на одних задачах.
Заказ и дизайн — оператор, 2026-07-13. Это первый внедрённый
экземпляр принципа A/B-eval механизма качества (бэклог
docs/RETRO_PATTERNS.md №12, урок Case 77/97: система может и
вредить — не предполагать, а мерить). Экзаменуемый «механизм» —
политика делегирования целиком; ось 8 SIBLING_MAP.

## Три плеча

Одна и та же модель, одна версия harness'а, одни задачи во всех
трёх. Полигон — песочница (свежий пустой проект), НЕ рабочие репо.

- **A — чистая модель:** пустой проект; без CLAUDE.md, без агентов,
  без спец-промпта.
- **B — наш деплой:** свежая установка РЕЛИЗНОГО шаблона тулкита
  (политика + агенты + журнал). Boot-налог шаблона честно входит в
  стоимость плеча (наш же тезис: boot-контекст платный, D-0038).
- **C — «хороший промпт» без инфраструктуры:** канонический промпт,
  дословно (слово оператора 2026-07-13):
  «Используя воркфлоу-подход, запускай субагентами Sonnet и Opus
  там, где это возможно».
  Записанное следствие: Haiku в промпте намеренно не назван — C
  изображает сильного пользователя без нашей политики; разница в
  дешёвой разведке (haiku-ярус) — одно из измеряемых отличий B от C.
  Смысл плеча: изолировать вклад ИНФРАСТРУКТУРЫ от вклада простого
  напоминания. C ≈ B — возим лишний вес; B > C — политика доказана
  против сильнейшей дешёвой альтернативы.

## Задачи

3–5 разноплановых, исполняемых (не виньетки). Ключевое требование
формы (слово оператора 2026-07-13): формулировка НАМЕРЕННО
НЕДОопределена — как реальный запрос пользователя («напиши мне
калькулятор»), с долей архитектурных решений, которые экзаменуемый
должен принять сам. Полностью специфицированная задача — это
builder-класс, «и Sonnet справится»: координационный слой
(декомпозиция, выбор архитектуры, спеки воркерам, приёмка) на ней
не экзаменуется. Именно недоопределённость заставляет плечо
показать, КАК оно превращает намерение в план и кому раздаёт части.

Состав: минимум одна задача «продукт с нуля» (недоопределённая,
архитектурная), одна разведка по чужому репозиторию, одна
доработка/ревью существующего кода; разноплановость важнее числа.

Требования:

1. НЕ из наших репозиториев и не решённые в них (контаминация).
2. Ключи приёмки пишутся ЗАРАНЕЕ, но на уровне НАМЕРЕНИЯ (что
   результат должен уметь, как это проверим), НЕ спеки реализации —
   спеку экзаменуемый обязан построить сам; ключи плечам не
   выдаются (дисциплина предрегистрации, ось 8).
3. Набор пинуется до первого прогона; замена задачи = строка в
   Runs log с причиной.

## Метрики (по убыванию несущести)

1. **Цена за ПРИНЯТУЮ задачу** — учётные цены D-0032, cache-aware,
   ВСЕ попытки и ретраи включены (Update Rule 4). Непринятая задача
   не даёт плечу очков, сколько бы ни сэкономила.
2. **Доля делегированной работы** — расход сайдчейнов / полный
   расход (cc_usage; меряется во всех плечах, журнала не требует).
3. **Эскалации/повторы** до приёмки.
4. **Скорость** (заказ оператора 2026-07-13): wall-clock задачи —
   первый..последний ts транскрипта сессии (UTC в транскриптах,
   перевод в локальное — при чтении).
5. Для B дополнительно: журнальная дисциплина соблюдена (события
   диспатчей/приёмок на месте) — иначе плечо мерило не нашу систему.
6. Качество сверх бинарной приёмки (заказ оператора 2026-07-13):
   вердикт фиксирует качественные различия плеч (тесты/ревью/
   безопасность/след), даже когда обе стороны «приняты» — цена без
   качества не читается.

Приёмка по DoD — Lead; спорные вердикты — оператор. Инструмент
учёта: tools/usage_report.py (per-session), сессии плеч разнесены
по проектам-песочницам и помечены в Runs log.

UI-приёмка (правило с 2026-07-14, прецедент прогона №1): задача,
чей результат — UI, принимается только ВОЖДЕНИЕМ UI (скриншоты/
запись «до/после» как witness-класс D-0052, паттерн Case);
статическое чтение кода недостаточно — визуальный дефект строки
выражения в C-T1 прошёл мимо кодовой приёмки Lead и был замечен
оператором по скрину (внешний контур F-11, второй раз за прогон).

## Честность замера

- Один прогон шумный: минимум — фиксировать разброс и помечать
  выводы «одиночный прогон»; норма — медиана 2–3 повторов на задачу
  («never trust one run», eval-план).
- Плечи прогоняются близко по времени (одна версия модели/harness).
- Порядок плеч на задаче варьируется (у оператора нет «любимого
  последнего плеча» с уточнённой формулировкой).
- Учётная честность: сессии экзамена живут в песочницах вне рабочих
  репо (атрибуция по проекту в cc_usage); окно экзамена помечается
  явно в notes ближайшей калибровки — синтетическая нагрузка не
  маскируется под рабочую (класс D-0075) и не искажает тренды
  чеков 11/18 молча. API-расход плеча B (если политика делегирует
  на API-контур) входит в стоимость плеча из requests.db по учётным
  ценам; отдельного «API-плеча» нет — на API-контуре нет носителя
  координатора (Pi гоняет воркеров), строить его ради экзамена —
  против Rule #1 (записано 2026-07-13, вопрос оператора).

## Набор №1 (пинуется 2026-07-13; ключи — уровень намерения, плечам не выдаются)

- **T1 «Калькулятор»** (архитектурная, с нуля). Текст плечу
  дословно: «Напиши мне калькулятор». Ключи приёмки: (1) работает и
  продемонстрирован (witness-класс: запуск/тесты); (2) контрольные
  выражения: 2+2*2=6 (приоритеты), скобки, деление на ноль
  обработано осмысленно, отрицательные, дробные; (3) архитектурные
  выборы (форма CLI/GUI/web; парсер vs eval, безопасность если
  eval) НАЗВАНЫ и обоснованы; (4) тесты либо явная демонстрация
  корректности.
- **T2 «Разведка click»** (чужой репозиторий). Текст: «В каталоге
  лежит клон репозитория click. Разберись, как в нём устроены
  группы команд и chain-режим, и подготовь отчёт с планом
  добавления алиасов команд: что менять, где, какие риски». Ключи:
  (1) названы реальные носители (Group/MultiCommand, click/core.py,
  путь resolve_command/get_command); (2) chain-семантика описана
  верно; (3) план алиасов реализуем (переопределение get_command /
  маппинг имён или эквивалент), файлы названы; (4) отчёт несёт след
  (куда смотрел). Приёмка сверяется по живому клону.
- **T3 «Починка todo»** (существующий код). Текст: «В каталоге
  маленький todo-менеджер. Пользователи жалуются: после удаления
  задачи номера съезжают и delete удаляет не то. Почини и сделай,
  чтобы этот класс багов больше не возвращался». Ключи: (1) баг
  назван по существу (позиционные номера как идентификаторы);
  (2) фикс — стабильные id или эквивалентный инвариант, не
  заплатка; (3) «класс закрыт»: регресс-тест/инвариант; (4) list/
  done/add не сломаны (прогон). Фикстура:
  docs/exam_fixtures/run1/todo_cli/ (мастер-копия, в песочницы
  копируется).
