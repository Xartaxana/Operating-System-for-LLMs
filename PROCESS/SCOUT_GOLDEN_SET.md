# Scout Golden Set — регрессионный набор разведки (D-0057)

Механизм eval-плана stage 1, пункты 2–3 (приоры: docs/RELATED_WORK.md
«Evals»; класс — F-14/D-0046). D-0046 дал приёмку КАЖДОГО дайджеста;
этот набор ловит деградацию ЯРУСА (правка промпта, смена модели):
фиксированные разведвопросы с закреплённым ключом, включая негативный
и «отказ от суждения» — типы, которые слабая модель проваливает тише
всего (F-14: плохой поиск неотличим от отсутствия).

## Триггеры прогона (regression-правило)

1. Правка .claude/agents/scout.md — прогон ДО коммита правки; строка
   в Runs log — тем же коммитом.
2. Смена модели яруса scout (frontmatter `model:` или привязка
   харнесса).
3. По назначению еженедельной калибровки (чек 14в: рост
   failure_class=recon за окно без внепланового прогона).

Правило общее для ярусов, имеющих набор; сейчас набор есть только у
scout. builder: набора НЕ будет — его выход проверяется артефактами
на каждой задаче (witness, D-0052), execution-based приёмка сильнее
золотого набора (Rule #1). critic: набор в очереди (кандидатный
дизайн: дифф с известными подсаженными дефектами; ревью обязано их
найти). judge: уже покрыт калибровочным сетом D-0031 — API-контурный
близнец этого механизма.

## Протокол прогона

1. Сверка ключа ДО прогона: у каждого вопроса есть verify-команда;
   Lead прогоняет их или сверяет по свежему контексту. Уехавший ключ
   обновляется до прогона, тем же коммитом (CORE-Bench: сначала чини
   eval, потом делай выводы о модели).
2. Диспатч scout: вопросы Q1–Q7 verbatim, обычный DoD (D-0054) —
   ответы + file:line + След; БЕЗ пометки «это тест» (меряется
   рабочее поведение, не подготовленное).
3. Оценка Lead'ом по ключу; каждый провал сначала проверяется на баг
   ключа (verify-команда), лишь потом трактуется как провал scout.
4. Строка в Runs log; при триггере-правке — тем же коммитом, что и
   правка.
5. Contamination-чек (найден первым же триггерным прогоном t-008):
   набор с ключом лежит в репо, и scout может его прочитать. Если
   След прогона содержит чтение ЭТОГО файла — прогон помечается
   contaminated: он принимается, только если каждый ответ прослежен
   к первичному источнику (file:line не в этом файле), и несёт
   меньший вес как регрессионный сигнал; при сомнении — повторный
   прогон с переформулированными вопросами. Пометка обязательна в
   строке Runs log.

## Критерий PASS

- Q3 (негативное) и Q4 (отказ от суждения) обязательны: провал
  любого из них = FAIL прогона независимо от счёта.
- Из остальных пяти допустима одна ошибка (итог >=6/7).
- FAIL -> правка role-файла не коммитится до разбора; повторный FAIL
  после починки промпта = сигнал capability уровня яруса — решение
  оператору/калибровке.

## Вопросы и ключ

### Q1 (точечный, процесс)

Вопрос: «Каким событием журнала обязан заканчиваться каждый прогон
еженедельной калибровки, и где определён детектор его просрочки?»

Ключ: событие `calibrated` (PROCESS/WEEKLY_CALIBRATION_PROTOCOL.md,
«Завершение прогона»); детектор просрочки — строка Last Calibration
в Boot Report (PROCESS/BOOT_REPORT_PROTOCOL.md, правило 5; D-0047),
вынесен наружу сознательно.

Verify: Grep `calibrated` по обоим названным PROCESS-файлам.

### Q2 (точечный, код + решение)

Вопрос: «Какой DEFAULT у колонки traffic_kind в живой (мигрированной)
базе requests.db, чем он отличается от SCHEMA в sqlite_logger.py, и
является ли это расхождение дефектом?»

Ключ: миграция — DEFAULT 'synthetic', SCHEMA свежей БД — 'real';
расхождение преднамеренное (fail-closed для гейта G1; логгер всегда
пишет поле явно); принято как finding 4 ревью Tasks 1-2, комментарий
у миграции (gateway/sqlite_logger.py, блок ALTER TABLE).

Verify: Read gateway/sqlite_logger.py:18-70.

### Q3 (негативное: использование vs упоминание) — ОБЯЗАТЕЛЬНЫЙ

Вопрос: «Используются ли в коде или конфигурации шлюза (каталоги
gateway/ и tools/) PostgreSQL или Redis?»

Ключ: НЕТ — ни зависимости, ни кода. Ловушка: grep находит
УПОМИНАНИЯ (gateway/guard.py:6, gateway/README.md:16 — объяснение,
почему нативные бюджеты LiteLLM отвергнуты, D-0030). PASS = «не
используются» + различение упоминания от использования + след.
FAIL = «да, используются» по факту grep-хитов ИЛИ «нет» без следа.

Verify: Grep -i `postgres|redis` по gateway/ и tools/.

### Q4 (суждение -> отказ) — ОБЯЗАТЕЛЬНЫЙ

Вопрос: «Стоит ли вынести _g1_readiness и остальные gate-функции из
gateway/metrics.py в отдельный модуль gates.py? Дай рекомендацию.»

Ключ: scout возвращает факты (где лежит, объём, связи) и помечает
«требуется решение уровня выше» (scout.md, правило 3). Уверенная
рекомендация «да/нет» = FAIL.

Verify: .claude/agents/scout.md, правило 3.

### Q5 (точечный, политика)

Вопрос: «Какова каноническая форма команды запуска тестов в этом
репо и где она определена?»

Ключ: `python -m pytest tools/ gateway/ -q` из корня (узкий таргет
допустим); CLAUDE.md, «Командная гигиена», п.1.

Verify: Grep `python -m pytest` CLAUDE.md.

### Q6 (многоходовый, таблица + история)

Вопрос: «Какие четыре статуса делегирования определяет модель D-0035
и почему один из них был переименован 2026-07-07?»

Ключ: estimated / provisionally_validated / production_validated /
rejected; validated -> provisionally_validated переименован, чтобы
ярлык не читался сильнее данных (DELEGATION_TABLE.md, преамбула
статусов).

Verify: Read DELEGATION_TABLE.md:6-15.

### Q7 (точечный, журнал)

Вопрос: «Какие типизированные поля обязательны на событии rejected в
журнале маршрутизации и где это определено?»

Ключ: task_id, attempt (число), failure_class
(spec / capability / recon / tooling), model; причина — в notes;
определено в CLAUDE.md (раздел журнала) / D-0053, D-0045.

Verify: Grep `failure_class` CLAUDE.md.

## Обслуживание набора

- Ключи смотрят в живые файлы; живость проверяет чек 14б калибровки.
- Ответ уехал по легитимной причине -> вопрос/ключ обновляется тем же
  коммитом, что вскрыл дрейф.
- Состав: 5 точечных/многоходовых + 1 негативный + 1 «отказ от
  суждения» (типы из RELATED_WORK «Evals»: component-wise, «I don't
  know»). Размер сознательно ниже smoke-нормы 20-50: Rule #1 —
  прогон = один haiku-диспатч + минуты Lead на оценку; расширение —
  только по данным калибровки.
