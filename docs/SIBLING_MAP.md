# Sibling Map — карта осей симметрии (D-0043)

Инструмент правила «чини класс, а не экземпляр»: закрывая дефект, обход
аналогичных мест делается ПО ЭТОЙ КАРТЕ, а не сканом репозитория.
Карта намеренно мала и грузится только в момент закрытия дефекта (не
при boot — D-0038). Порядок работы:

1. Найди ось (оси), на которых лежит дефект — см. список ниже.
2. Проверь/почини парные места оси точечно; что не чинится сейчас —
   явно в очередь (CURRENT_CONTEXT.md) или журнал.
3. Класс шире карты или ось не находится → диспатчь scout (haiku) на
   точечный поиск с конкретным вопросом. Lead/critic НЕ перечитывают
   репозиторий руками (правило 1).
4. Появилась новая симметрия (новый деплой, новый парный документ,
   новая пара код/учёт) → добавь ось сюда В ТОМ ЖЕ коммите — это
   часть шага 3 D-0043.

critic проверяет классовую полноту фикса ПО КАРТЕ: «покрыты ли оси,
на которых лежит дефект» — ограниченный вопрос по диффу и карте, не
по всему коду.

## Проверка самой карты (D-0048)

Карта — инструмент D-0043 и сама нуждается во внешней проверке;
собой её проверить нельзя (тот же класс, что F-15):

1. **Живость** — еженедельная калибровка (чек 12 протокола)
   проверяет, что каждый конкретный путь, названный в карте,
   существует в обоих репо. Переименовал/удалил файл — правь ось
   тем же коммитом.
2. **Полнота — по рецидивам.** Дефект класса, уже чинившегося по
   карте, всплыл в месте, куда карта не указывала → это находка О
   КАРТЕ (пропущенная или неверная ось), а не только о дефекте:
   фиксируется в FINDINGS, ось правится тем же фиксом.
3. **Рост.** Число строк карты считает калибровка (чек 10). Оси
   умирают симметрично рождению: симметрия исчезла (деплой закрыт,
   механизм удалён) → ось удаляется ТЕМ ЖЕ коммитом, что убил
   симметрию.

## Поддержание карты из второго репозитория

Карта одна — здесь (две карты сами стали бы расходящимися
«собратьями»). Сессия AO3, нашедшая новую симметрию: (а) если может —
отдельным коммитом в ЭТОТ репозиторий, сразу; (б) если нет — явная
строка в свой HANDOFF/журнал «новая ось для SIBLING_MAP: <что>», и
Lead OS-репо вносит её при следующем касании. Молчаливо не вносить
нельзя — это тот же класс нарушения, что и молчаливый пропуск (F-9).
