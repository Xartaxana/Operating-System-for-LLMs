## Часть 1. Что индустрия использует для защиты от галлюцинаций и «инициативы» ИИ (не промптами)

Общая философия (Anthropic формулирует её явно): модель — недоверенный компонент;
вкладываться нужно в детерминированную инфраструктуру (границы, перехват, откат),
а не в «уговаривание» модели промптами.

1. **Система разрешений на вызовы инструментов (permission gates / allowlists).**
   Каждый tool call проходит слой правил ДО исполнения: allowlist/denylist по инструменту,
   пути, шаблону команды. У Claude Code — permission modes; у OpenAI Agents SDK — approvals.
   Anthropic обнаружила, что пользователи одобряют ~93% запросов (усталость от подтверждений),
   и сделала auto mode — классификатор, сам одобряющий безопасные действия.

2. **Хуки / middleware вокруг tool calls.** Детерминированные скрипты (не промпты),
   перехватывающие вызов до/после исполнения — как auth-middleware в HTTP. Могут блокировать,
   менять, запускать линтер/сканер. Правило из практики: хуки маленькие и детерминированные.

3. **Песочницы на уровне ОС.** bubblewrap (Linux), Seatbelt (macOS), контейнеры/VM.
   Ограничение ФС (только рабочая директория) и сети (egress только к разрешённым доменам).
   Внутри границы агент свободен; вырваться не может физически («blast radius»).

4. **Constrained decoding / structured outputs.** Сэмплер физически не может сгенерировать
   токен вне JSON-схемы/грамматики (Outlines, XGrammar, LM Format Enforcer; OpenAI Structured
   Outputs; валидация схем tool calls у Anthropic). Убивает выдуманные поля и битый JSON
   (~65% ошибок схем). Известный кейс: снижение ошибок пост-обработки с 32% до 0.4%.
   Нюанс: жёсткий формат может снижать качество рассуждений — думать свободно, схему
   требовать на финальном ответе.

5. **Validate-and-retry циклы.** Выход прогоняется через валидатор (JSON Schema, Pydantic,
   проверка существования упомянутых файлов/ID); при провале — автоматический повтор
   с текстом ошибки. Вывод не принимается, пока не пройдёт проверку.

6. **Guardrails-фреймворки с tripwire.** OpenAI Guardrails: input-/output-/tool-guardrails;
   при срабатывании исполнение мгновенно останавливается. Из коробки: детекторы галлюцинаций,
   PII, джейлбрейков, off-topic, фильтрация URL. Аналоги: NeMo Guardrails, Guardrails AI.

7. **LLM-как-судья, гейтящий исполнение.** Вторая модель — чекпоинт ДО выполнения действия /
   выдачи ответа. Продвинутый вариант: панель «скептиков», каждый пытается опровергнуть;
   проходит пережившее большинство.

8. **Dual-LLM паттерн (карантин недоверенного контента).** «Привилегированная» модель
   планирует и зовёт инструменты, но не видит сырой недоверенный текст; «карантинная»
   читает недоверенное, но без инструментов. Между ними — только типизированные структуры.
   Плюс правило харнеса: контент из tool results = данные, не команды.

9. **Human-in-the-loop по категориям действий.** Чтение свободно; запись по правилам;
   необратимое/внешнее — только с явным подтверждением человека, не переносимым между
   действиями. Это состояние харнеса, а не просьба в промпте.

10. **Заземление во внешнюю истину.** Компилятор, тесты, линтер, CI как валидатор;
    «зелёный» статус — единственный критерий готовности. RAG с обязательными цитатами.

11. **Лимиты ресурсов и отката.** Бюджеты токенов/стоимости, таймауты, лимит итераций,
    изоляция в git-worktree/бранче. Git — слой undo для агента.

12. **Логирование и трассировка.** Каждый tool call, аргументы, решение guardrail'а —
    в аудит-лог (OpenTelemetry, LangSmith, tracing в Agents SDK).

## Часть 2. Приёмы валидации в vibe-engineer (github.com/ismailsaleekh/vibe-engineer)

Система исходит из того, что модели нельзя верить на слово вообще: каждое утверждение
либо подтверждено исполняемым доказательством, либо не считается.

1. **Трёхуровневые «механические гейты» (p0/p1/p2)** — пакет `@vibe-engineer/mechanical-gates`.
   p0: архитектурные границы зависимостей, чистота доменов, публичные поверхности пакетов,
   границы тестов, guard'ы конфигов. p1: review-scanner, строгость схем контрактов,
   анти-паттерны тестов, quality-ratchet. p2: code smells. Правила — в JSON в корне репо
   (mechanical-boundaries.json и др.).

2. **Структурные подписи по AST вместо текстовых паттернов.** Review-scanner парсит
   TypeScript AST и считает SHA-256 «структурную подпись». Исключения в
   mechanical-escape-allowlist.json привязаны к: (а) подписи кода, (б) документу-решению
   docs/decisions/DL-xx.md, (в) SHA-256 этого документа. Переименование не обходит гейт
   (коды STRUCTURAL_EVASION, WORDING_FALSE_POSITIVE); изменение кода или обоснования
   рвёт хеш. Каждая «дырка» криптографически заморожена вместе со своим оправданием.

3. **Fail-closed тотально.** Неизвестный CLI-флаг → отказ; объявленное, но
   незарегистрированное quality-семейство → wiring-integrity гейт блокирует ВСЁ до
   остальных проверок; неподдерживаемая поверхность харнеса → типизированный отказ,
   не тихий fallback.

4. **Evidence Packets с криптопривязкой (evidence-v2).** «Никогда не принимает пересказ
   вызывающего вместо пакета — каждый значимый байт перечитывается из файла,
   пере-хешируется и заново привязывается к runner/task/scope». Коды ошибок на подмену,
   path traversal, повышение уровня доверия (TRUST_RELABEL_FORBIDDEN,
   PROOF_BOUNDARY_ESCALATION), секреты в evidence.

5. **Witnesses — исполняемые доказательства свойств.** Вместо утверждения о структуре —
   скрипт, проверяющий её исполнением (workspace-surface-witness.mjs в package.json).
   Свойство истинно, пока witness зелёный.

6. **Песочница ревьюера через Node preload (reviewer-process-guard.js).** Перехват fs-API
   дочернего процесса: читать — ограниченный контекст, писать — ТОЛЬКО в один точный файл
   отчёта; иначе маркер нарушения + REVIEWER_PROCESS_ESCAPE_DETECTED.

7. **LLM-ревьюер физически read-only + схема.** Runner вызывает:
   `claude -p --output-format json --json-schema '<схема>' --permission-mode plan
   --allowedTools 'Read,Grep,Glob,Bash(git diff:*),...' --disallowedTools 'Edit,Write'`.
   Fail-closed на невалидный JSON, несовпадение схемы, ненулевой exit, отказ в правах.

8. **Версионированная цепочка артефактов.** Work Brief 2.0.0 → Implementation Plan 2.0.0 →
   Build Result 2.0.0 → Ship Packet 2.0.0; каждый этап — JSON-артефакт со схемой,
   version-gate (старая версия → MIGRATION_REQUIRED) и дайджестом. Между этапами —
   только типизированные артефакты, не свободный текст.

9. **Финальный агрегатор Definition-of-Done (final-dod-v2).** Единственная точка «готово»:
   закрытие findings ревью, покрытие профилей, product proof, валидность evidence.
   Object.freeze + WeakMap authority — статус нельзя подделать изнутри процесса.

10. **Таксономия «детерминированное vs мнение модели».** Вердикты через модель — класс
    `model_advisory`, «никогда не deterministic», на уровне типов.

11. **Гейт публикации + человек в петле.** VIBE_ENGINEER_RELEASE_APPROVED=true +
    --confirm-publish + release:check + install-smoke. Workflow-скиллы
    disable-model-invocation / user-invoked — модель не запускает себя на следующий этап.

Итог: почти весь канонический набор Части 1, плюс три редкие идеи — evidence-first
(утверждение без перечитанного доказательства = ложь), AST-подписи исключений,
witnesses. По сути перенос идей формальной верификации и supply-chain security
(attestations, provenance) на agentic-разработку.

## Часть 3. Что из этого внедрять в Operating-System-for-LLMs

### Уже есть (импортировать не нужно)

- Детерминированные хуки-гейты на всех событиях (settings.json: dispatch_gate,
  hygiene_gate, dod_track/dod_gate, main_gate, journal_echo, critic_snapshot).
- DoD-инвариант «после последней правки — зелёный прогон» (dod_gate для субагентов,
  main_gate t-159 для координатора).
- Critic как обязательный гейт приёмки (R3), двухслойный вход.
- Append-only журнал JSONL с валидатором (journal_validator, pre-commit).
- Fail-closed (mechanism_gate п.5, F-7).
- Самозащита enforcement-цепочки (D-0069; mechanism_gate п.6 — правка гейта не обходит гейт).
- Разделение «код фиксирует ПОРЯДОК событий, смысл СУДИТ ярус выше» (D-0063) —
  прямой аналог model_advisory.
- Эскалация, human-in-the-loop, приёмка у Lead (D-0037).

### Рекомендовано внедрить (по приоритету)

**№1. Перекрёстная проверка witness против dod_track (принцип evidence-v2). Самое ценное.**
Дыра: journal_validator проверяет только «witness — непустая строка»
(tools/journal_validator.py, проверка 7) — witness остаётся самодекларацией; вставленный
из повествования или устаревший текст проходит. При этом dod_track.py (PostToolUse) уже
детерминированно записывает реальные прогоны. Механизм: валидатор/гейт требует, чтобы
witness ссылался на реально наблюдавшийся хуком прогон — совпадение команды и
хеша/подстроки вывода из трека. Обе половины механизма уже существуют; тот же класс
находки, что main_gate t-159 («координатор мог сам — никто не ловил»).

**№2. Wiring-integrity гейт (из run-quality.mjs vibe-engineer). Почти бесплатен.**
Дыра: хуки в settings.json молча умирают (python недоступен, скрипт переименован,
исключение при fail-open); если core.hooksPath не выставлен — mechanism_gate и
journal_validator вообще не запускаются, что неотличимо от «всё прошло» (это твой же F-7).
Механизм: session_context.py на SessionStart проверяет — hooksPath выставлен, каждый файл
из settings.json существует и импортируется, python в PATH; расхождение → громкое
предупреждение/блок.

**№3. Привязка постоянных исключений к решению через хеш (escape-allowlist).**
Дыра: постоянные концессии-правила живут текстом в CLAUDE.md/DECISIONS_FULL без привязки;
класс «обоснование молча уехало от правила» не ловится. Механизм: машинно-читаемый список
исключений, каждая запись несёт D-номер + SHA-256 цитируемого фрагмента решения; гейт
проверяет актуальность. Приоритет средний (DECISIONS_FULL append-ориентирован, дрейф
менее вероятен).

**№4. Схемный вывод critic'а + физически read-only запуск.**
Дыра: вердикт critic'а — свободный текст; «вердикт есть/полон» судится Lead'ом на глаз.
Механизм: вызов ревьюера с принудительной JSON-схемой вердикта (verdict / blockers /
class-completeness по SIBLING_MAP) и read-only набором инструментов
(--disallowedTools Edit,Write), fail-closed на отклонение схемы — по образцу runner'а
vibe-engineer (см. Часть 2, п.7).

### Не тащить

- p0-гейты архитектурных границ и AST-подписи — заточены под TS-монорепу с графом пакетов;
  в OS плоский набор Python-инструментов, границы охранять не от чего. mechanism_gate уже
  прошёл грабли текстового сканирования (F-A/F-B) и решил их сужением области.
- Node preload-песочница ревьюера — роль уже играют allowed-tools субагентов + D-0069.
- Release-гейтинг и цепочка Work Brief → Ship Packet — публикаций нет; роль цепочки
  выполняют журнал + DAG в docs/tasks; дублировать схему — оверхед.
