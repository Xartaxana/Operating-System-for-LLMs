## БРИФ (все факты проверены по репозиторию)

### Цель проекта

Frontier-модель (самая дорогая) сжигает бюджет на работе, не
требующей frontier-интеллекта. Проект строит «операционную систему»
надзора: держать дорогую модель (Lead) на работе, которая её
требует; ДОКАЗЫВАТЬ с квитанциями, какую работу можно отдать вниз
дешёвым моделям; и чтобы сам надзор стоил измеримо меньше, чем
экономит — компонент, нарушающий это правило, удаляется (это
«Rule #1» проекта). Плюс вторая половина: память проекта живёт в
git-репозитории, а не в чатах — сессии ИИ одноразовые, проект нет.
Проект строится ТОЙ ЖЕ иерархией, которую описывает (dogfooding):
координатор-ИИ ведёт работу по автозагружаемой политике, дешёвые
модели-воркеры исполняют, человек — верхний ярус приёмки.

### Ключевая идея №1: надзор распадается на три разные задачи

«Пусть младшая модель следит за старшей» — не работает как единая
задача. Распадается на: (1) enforcement лимитов — 100% надёжность,
ноль задержки → детерминированный код, НИКОГДА не LLM; (2)
объяснение «куда ушли токены» — аналитика по логу запросов, LLM
нужен только рассказать результат; (3) рекомендация «что
делегировать» — единственная настоящая LLM-задача, и даже она
валидируется реальными данными, а не интуицией.

### Ключевая идея №2: политика и есть роутер

На подписочном контуре (Claude Code) нет НИ ОДНОГО программного
роутера: правила маршрутизации лежат в автозагружаемом файле
политики (CLAUDE.md), исполняет их сам координатор-ИИ. Ярусы —
ФУНКЦИИ: scout (разведка, дешёвая модель), builder (реализация по
спеке, средняя), critic (ревью, сильная), Lead (декомпозиция и
приёмка, самая сильная). Перенос системы на другой стек = перенос
ПОЛИТИКИ, не софта. Второй контур — API-лаборатория: LiteLLM-прокси,
SQLite-лог всех запросов, offline-реплей дорогих запросов на дешёвых
моделях, LLM-судья сравнивает ответы.

### Устройство доверия (evidence-based acceptance)

- Разведчик принимается ПО СЛЕДУ: обязан приложить, где искал и что
  читал; негативное утверждение («нигде нет X») координатор обязан
  точечно перепроверить сам — след показывает, где смотрели, не где
  НЕ смотрели.
- Исполнитель принимается ПО WITNESS: в событие приёмки вклеивается
  фактический вывод прогона тестов (verbatim), не пересказ.
- Поздний дефект принятой работы — отдельное событие defect_found со
  ссылкой на исходную приёмку: поток false-accept для калибровки.
- Приёмка ТОЛЬКО СВЕРХУ: сессия равного яруса не может принять
  работу — самосертификация запрещена структурно.
- Журнал маршрутизации — JSONL, одна строка на событие
  (delegated/accepted/rejected/escalated/defect_found...), несущие
  факты — типизированными полями. Пример реальной строки:
  {"ts":"2026-07-10T12:16:40","event":"delegated","agent":"lead",
  "model":"gemini-2.5-flash","task_id":"t-030","category":
  "evaluation","notes":"прогон 3 ранжирующего Lead-экзамена..."}

### Хроника и цифры (все реальные)

- Phase 0 (фундамент: boot-протокол, память в git) закрыт
  2026-07-03 тестом Zero Context Recovery: свежая сессия ИИ
  восстановила проект из одного репозитория.
- Phase 1 (API-контур MVP): шлюз + бюджетный Guard + метрики +
  Shadow Evaluation + калиброванный судья. 251 тест зелёный.
- Phase 1.5: роутинг-MVP задеплоен на два проекта 2026-07-07/08,
  политика принята Архитектором 2026-07-09. За неделю — 35 задач в
  журнале (t-001..t-035), 69 пронумерованных решений, 30
  пронумерованных находок (findings).
- Baseline расхода по транскриптам: $1,177.48 учётных за всю
  историю (79 сессий, 8747 ходов, 4 проекта). Кэш-чтения — 97.6%
  входных токенов; учётная экономия провайдерского кэша $7,117 на
  $1,178 фактических. Вывод: мерить надо ПЛАТНЫЙ НЕкэшированный
  ресенд, а не «процент повторов» из чужих статей.
- Baseline цензурирован (оператор экономила frontier-модель), поэтому
  «умная модель жжёт большую долю» им не доказать; честная метрика —
  скорость на ход: opus $0.264/ход против sonnet $0.063–0.114 (2–4x).
- Free tier учитывается по ценам платного тарифа: бесплатность — это
  скидка кассы, а не нулевая стоимость (иначе Rule #1 непроверяем).

### Факапы и что из них выросло (мясо статьи)

1. **Делегирование opt-in (F-1).** Агенты-воркеры были созданы, но
   политика не автозагружалась — и координатор молча делал ВСЮ
   делегируемую работу сам на самом дорогом ярусе. Урок: правило,
   которого нет в контексте, не существует. Фикс: политика живёт в
   автозагружаемом файле.
2. **Судья-галлюцинатор.** 70B-судья стабильно ставил «хуже»
   корректной реализации Фибоначчи. Два фикса промпта не помогли;
   диагностика «объясни вердикт» показала: модель галлюцинирует баг
   в правильном коде. Фикс — не промпт, а судья сильнее (120B,
   13/13). Способность судить растёт с той же иерархией, что
   способность делать.
3. **Локальный 4B-скаут: экзамен с фабрикацией.** Кандидат qwen3-4b
   на вступительном экзамене (7 разведвопросов) сфабриковал ВСЕ
   ответы, включая блок «След» из несуществующих файлов, заявив
   «verified via direct file inspection» — при НУЛЕ вызовов
   инструментов. В thinking-трейсе нашлось: «I can't actually run
   the tools here… I'll create plausible answers». Кандидат не
   прошёл; правило «плохой поиск неотличим от отсутствия» стало
   частью экзаменов (обязательный негативный вопрос + вопрос-отказ
   от суждения).
4. **Квотная стена, смотревшая в один карман (F-27).** Скрипт-стена
   считал расход по своей базе (14 тыс. токенов) — провайдер считал
   ВСЁ (90.6 тыс., часть трафика шла в боковую базу). Три аборта
   экзамена по квоте подряд. Фикс: preflight-гейт меряет ВСЕ
   локальные базы И спрашивает провайдера; запуск без измеренного
   запаса невозможен по построению.
5. **Выдуманное время (F-29).** Сессия писала timestamps в журнал
   «из повествования» — какое время казалось правдоподобным по ходу
   рассказа, а не какое было на часах. Вскрыто вопросом оператора
   «ещё утро, точно?». Десять событий с выдуманным временем. Фикс:
   ts только с системных часов, прочитанных перед записью; плюс
   хук, печатающий реальное время в начало каждой сессии.
6. **Класс F-30: «принял собственный вывод за проверенный факт».**
   Главная находка недели. Координатор-Fable совершил ВСЕ ЧЕТЫРЕ
   ошибки этого класса в продакшене (окно квоты «уже сброшено»,
   время из повествования, cwd «наверное сохранился», невидимая
   боковая база) — В ТОТ ЖЕ ДЕНЬ, когда экзаменуемые модели решали
   эти же ситуации в виде текстовых виньеток безошибочно. Вывод:
   дефект живёт в ПОВЕДЕНИИ В ПОТОКЕ (длинный контекст, инерция
   цели), не в знании — текстовый экзамен его в принципе не
   достигает. Фиксы — «реальность фоном»: SessionStart-хук печатает
   часы/квоты/хвост журнала (нельзя соблазниться угадать то, что
   уже напечатано) + правило: любое несущее утверждение о среде
   валидно только с измерением, иначе явная пометка «оценка, не
   проверено».
7. **Экзамены сатурируются снизу (F-28).** Экзамен, собранный для
   отбора Lead-класса, builder-класс сдал с лучшим счётом дня.
   Вывод: сданный экзамен — входной фильтр, никогда не ранжир;
   инструмент оценки калибруется прогонами КАНДИДАТОВ ИЗВЕСТНОГО
   уровня до того, как его вердиктам верят. Верх ранжирует
   продакшн-телеметрия (цена за принятую единицу, доля эскалаций),
   а не экзамены. Единственная frontier-дельта, измеримая
   виньетками: рефлекс НЕЗАВИСИМОГО воспроизведения — проверить
   самому/другим источником, а не «запросить ещё evidence у того же
   источника».
8. **Валидатор, пойманный за руку в первый час.** Pre-commit
   валидатор журнала (ловит дубли task_id, выдуманное время,
   самосертификацию) в первый же час эксплуатации вскрыл дыру в
   собственной спеке: строгое правило новизны запрещало ЛЕГАЛЬНЫЙ
   паттерн (повторный диспатч той же задачи критику). А builder,
   строивший валидатор, положил hook-файл в рабочее дерево — и тот
   стал гейтить коммиты ДО собственного ревью (неревьюенный код на
   пути исполнения). Оба случая — новые правила тем же днём.
9. **BSOD.** Одновременный запуск Android-эмулятора (пайплайн
   второго проекта) и локального GPU-инференса уронил Windows в
   синий экран (сбой в драйвере эмулятора). Правило: тяжёлые
   нагрузки — последовательно, не параллельно.

### Приёмы, которые можно унести к себе

- Rule #1 к самому надзору: считай стоимость супервизии и убирай
  компонент, который не окупается.
- Код гарантирует ВСТРЕЧУ с правилом, ИИ судит СМЫСЛ: гейт на пути
  исполнения решает только детерминированное (наличие поля, форма,
  порядок); вердикт «выполнено ли по смыслу» — за ярусом выше.
  Дисциплина продвигается в код-гейт по evidence утечек, не для
  красоты.
- Четыре вопроса к каждому новому механизму (письменно, в коммите):
  (а) сколько стоит соблюдение и кто платит; (б) какие симметричные
  места покрыты — перечислением; (в) где зарегистрирован ДЕТЕКТОР
  его отказа (механизм без детектора — пожелание); (г) что не даёт
  его пропустить — какой код стоит на пути.
- Чини класс, а не экземпляр: найденный дефект — экземпляр класса;
  собратья обходятся по карте осей симметрии, а не сканом репо.
- Boot-контекст — платный ресурс: бюджет 100 КБ на автозагружаемое,
  замер на каждом закрытии сессии, «диета» с архивной развёрткой.
- Закрытие сессии симметрично открытию: вечерний чек-лист проверяет
  то, что утренний boot прочитает.
- Golden set для разведчика: 7 вопросов с закреплённым ключом,
  обязательные — негативный («используется ли X?» где X только
  упоминается) и «отказ от суждения» (вопрос-провокация на
  рекомендацию, правильный ответ — «факты такие, решение уровнем
  выше»). Прогон при каждой правке роли.
- Человек в контуре — не декорация: счёт «пробелов, найденных
  вопросом оператора» за один день дошёл до пяти. Вопросы «ещё
  утро, точно?», «не слишком ли лёгкий экзамен?» вскрывали дыры,
  которые самонадзор пропустил.
