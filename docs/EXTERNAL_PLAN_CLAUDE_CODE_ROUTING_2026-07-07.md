## 1. Проблема

Самая умная модель (Fable) используется для всего подряд — и для архитектурных
решений, и для рутины (поиск по файлам, чтение, механические правки). Рутина
съедает большую часть токенов, не требуя интеллекта. Расход непрозрачен:
лимит заканчивается «внезапно», и непонятно, на что именно он ушёл.

## 2. Целевая картина

Умная модель работает **только** как архитектор и надзор: декомпозирует задачу,
пишет ТЗ, принимает результат. Исполнение уходит субагентам на дешёвых моделях.
Каждое делегирование журналируется. Раз в день/неделю генерируется отчёт:
сколько токенов, на какие модели, на какие задачи и почему.

## 3. Принятые решения (зафиксированы 2026-07-07)

| Вопрос | Решение |
|---|---|
| Источник лимитов | Подписка Claude **и** API-ключ — нужен сводный учёт |
| Область | Сначала оптимизируем работу в Claude Code, затем выносим в отдельную систему |
| Автономия роутинга | Полностью автоматический, по правилам из конфига |
| Отчётность | Дневной/недельный отчёт + журнал решений роутинга |

## 4. Архитектура

### 4.1 Уровни задач → модели

| Уровень | Тип задач | Модель |
|---|---|---|
| T0 — механика | Поиск по коду, чтение и сводка файлов, массовые правки по шаблону, boilerplate, форматирование | Haiku |
| T1 — стандартная разработка | Реализация по чёткому ТЗ, тесты, багфиксы с известной причиной, рефакторинг | Sonnet |
| T2 — сложное исполнение | Отладка неясных багов, код-ревью, дизайн отдельных модулей | Opus |
| T3 — архитектура и надзор | Декомпозиция, ТЗ, принятие решений, приёмка результатов, общение с пользователем | Fable (главная сессия) |

Правило эскалации: если младшая модель не справилась (результат не прошёл
приёмку), задача поднимается на уровень выше, а факт эскалации пишется в журнал —
это сырьё для калибровки правил в Фазе 3.

### 4.2 Компоненты

1. **Субагенты** (`.claude/agents/*.md`, у каждого своя модель в frontmatter):
   - `scout` (haiku) — поиск, чтение, сбор контекста; возвращает сжатую выжимку;
   - `builder` (sonnet) — реализация по ТЗ, написание тестов;
   - `critic` (opus) — ревью кода, сложная отладка.
   Субагенты работают в собственном контексте — контекст главной сессии не раздувается.
2. **Политика роутинга** (`CLAUDE.md` проекта): главная модель обязана
   декомпозировать и делегировать, а не делать рутину сама; ТЗ субагенту — компактное;
   от субагента требуется краткий отчёт, а не дамп файлов.
3. **Журнал роутинга** (`logs/routing-log.jsonl`): хук на запуск субагентов пишет
   запись — время, суть задачи, выбранная модель, обоснование, итог (принято/эскалация).
4. **Учёт расхода** (`tools/usage_report.py`): парсит транскрипты Claude Code
   (`~/.claude/projects/**/*.jsonl` — там есть модель и число токенов каждого
   сообщения), соединяет с журналом роутинга, пишет отчёты в `reports/`.
5. **Планировщик**: ежедневная генерация отчёта (Windows Task Scheduler).
6. **API-сторона** (Фаза 4): учёт расхода по API-ключам через Anthropic Usage API,
   сведение с подписочным расходом в один отчёт.

### 4.3 Поток данных

```
Пользователь → Fable (архитектор): декомпозиция, ТЗ
                 │ делегирует по правилам T0–T2
                 ▼
        Субагенты (Haiku / Sonnet / Opus) → сжатый отчёт → приёмка Fable
                 │
        хук пишет logs/routing-log.jsonl
                 │
Транскрипты сессий (токены по моделям) ──┐
                 │                        ▼
                 └────────────→ tools/usage_report.py → reports/usage-YYYY-MM-DD.md
                                                        reports/weekly-YYYY-WW.md
```

## 5. Фазы

### Фаза 0 — Наблюдаемость (первый шаг; нельзя оптимизировать то, что не видно)
- Скрипт `tools/usage_report.py`: расход токенов по дням / моделям / сессиям / проектам
  из транскриптов Claude Code.
- Хук журналирования делегирований в `.claude/settings.json`.
- Первый отчёт-baseline: как расходуются лимиты сейчас, до роутинга.

**Готово, когда:** можно за одну команду получить отчёт «вчерашний расход по моделям».

### Фаза 1 — Роутинг в Claude Code
- Определения субагентов `scout` / `builder` / `critic` с моделями.
- `CLAUDE.md` с политикой роутинга и правилом эскалации.

**Готово, когда:** типовая задача проходит цикл
«декомпозиция (Fable) → исполнение (Haiku/Sonnet) → приёмка (Fable)»,
и журнал роутинга заполняется автоматически.

### Фаза 2 — Автоматические отчёты и предупреждения
- Ежедневная генерация отчёта по расписанию; недельный сводный отчёт.
- В отчёте: доля токенов по моделям, топ-сессии по расходу, число эскалаций,
  тренд относительно прошлой недели.
- Предупреждение при аномальном темпе расхода старших моделей.

**Готово, когда:** отчёты появляются без ручного запуска и отвечают на вопрос
«на что ушли лимиты и почему».

### Фаза 3 — Калибровка правил
- Еженедельный разбор журнала: где дешёвая модель справилась, где нет.
- Правки таблицы T0–T2 по фактам, а не по ощущениям.
- Метрики: доля токенов Fable+Opus в общем расходе (цель — устойчивое снижение
  против baseline), частота эскалаций (если растёт — правила слишком агрессивны).

### Фаза 4 — Вынос в отдельную систему
- По накопленным правилам и данным — самостоятельный роутер поверх API
  (правила классификации из Фазы 3 становятся спецификацией).
- Подключение учёта API-ключей (Usage API) к общему отчёту.
- agent-dispatch для делегирования между проектами.

## 6. Метрики успеха

1. **Доля старших моделей**: % токенов Fable+Opus от общего расхода — снижается против baseline Фазы 0.
2. **Частота эскалаций**: сколько задач пришлось переделывать старшей моделью — остаётся низкой.
3. **Прозрачность**: на любой день есть ответ «на что ушли лимиты» без ручного разбора.

## 7. Риски и честные ограничения

- **Оркестратор тоже тратит**: чтение отчётов субагентов расходует токены Fable.
  Смягчение: требовать от субагентов сжатые отчёты, не дампы.
- **Плохое ТЗ = двойной расход**: если Fable поставит нечёткую задачу, дешёвая
  модель сделает не то, и переделка съест экономию. Журнал эскалаций покажет,
  где проходит граница возможностей младших моделей.
- **Точный остаток недельного лимита подписки программно не публикуется**:
  оцениваем собственным учётом токенов; точное значение — вручную через `/usage`.
- **Классификация должна быть дешёвой**: правила в конфиге, а не отдельный
  LLM-вызов на каждое решение о роутинге.

## 8. Следующее действие

Фаза 0: написать `tools/usage_report.py` и настроить хук журнала роутинга,
получить baseline-отчёт за последнюю неделю.
