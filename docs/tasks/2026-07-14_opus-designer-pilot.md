# Пилот «Opus-дизайнер»: спека ярусом ниже (2026-07-14 → разбор на калибровке ~07-18)

Слово оператора 2026-07-14, после прогона №1 экзамена экономии
(воркфлоу плеча C отдал design-фазу Opus: «может нам тоже так
выгоднее? Fable составляет промт с DoD, а точную спеку пишет уже
Opus»). Гипотеза: точная спека по интент-брифу Fable, развёрнутая
Opus, вдвое дешевле за токен и не тратит дефицитный Lead-контекст
(D-0040) при не худшем качестве. Таблица делегирования держит
вопрос строкой «Task decomposition + spec writing → strongest
available tier» (estimated) — пилот собирает evidence для Update
Rule 1. Главный известный риск — МОЛЧАЛИВОЕ интерпретационное
решение спекописателя (прецедент C-T1: Opus-спека выбрала
архитектуру, не спросив; урок вердикт-ревизии оператора) — лечится
обязательным блоком развилок ниже.

## Схема (экспериментальное дополнение правила 2 политики; пометка там)

1. **Fable — интент-бриф**: намерение, DoD уровня результата,
   границы/non-goals, УЖЕ принятые интерпретационные решения.
2. **Диспатч дизайнеру** («opus: спека <тема>»; general-purpose +
   model=opus — отдельный role-файл до итогов пилота НЕ заводится,
   чтобы не открывать оси 3/8 под эксперимент; решение записано
   здесь): развернуть бриф в точную спеку builder-диспатча по форме
   правила 11 (критерии приёмки + проверочный прогон + манифест).
   ОБЯЗАТЕЛЬНЫЙ блок «Развилки»: (а) «закрыл сам» — списком, с
   обоснованием каждого; (б) «требуют слова Lead/оператора» —
   вопросами. Спека без блока развилок возвращается как DoD-less
   (правило 11), решать развилки молча запрещено.
3. **Fable — приёмка спеки**: сверка с брифом + вердикт по каждой
   развилке из «закрыл сам» (перевёрнутая развилка — счётчик
   метрики 4); rejected → failure_class=spec на дизайнере.
4. Дальше штатно: builder по принятой спеке, приёмка по witness.

Декомпозиция, механизмы, политика — ВНЕ пилота (D-0037: authority
остаётся у Lead; делегируется ЧЕРНОВИК спеки, не решение о ней).

## Журнальная конвенция

`agent="designer"`, `model="opus"`, `category="spec"`, обычные
delegated/accepted/rejected с task_id. Валидатор: неизвестный agent
вне матрицы D-0058 (сверено кодом journal_validator, строка 208);
приёмка сверху выполняется фактом fable>opus. Лейбл диспатча —
«opus: спека …» (правило 7).

## Объём и чередование (пинуется до первого диспатча)

N≈5 builder-задач с нетривиальной спекой до калибровки ~07-18.
Чередование по порядку поступления: 1-я, 3-я, 5-я — спека Fable
(контрольная группа), 2-я, 4-я — через дизайнера. Однострочные
правки (класс «меньше диспатча») в счёт не идут.

## Метрики (пинуются сейчас; счёт — разбор пилота на калибровке ~07-18)

1. failure_class=spec rate builder-задач по opus-спекам vs
   fable-спекам окна.
2. Цена спеки: токены дизайнера + Fable-приёмка против Fable-спеки
   целиком (cc_usage, после t-094 — числа честные).
3. Поздние дефекты (defect_found, ref=) по обеим группам.
4. Развилки: вернул вопросами / закрыл сам / из «закрытых сам»
   перевёрнуто Fable на приёмке — прокси interpretation drift
   (урок C-T1).
5. rejected-спеки на приёмке Fable.

Решение двигает ТОЛЬКО калибровка (Update Rule 1): положительный
итог → мост словаря D-0062 (новая функция designer) + role-файл +
строка таблицы provisionally_validated; отрицательный → «спека
остаётся Lead», пилот закрывается записью здесь и снятием пометки
из правила 2.
