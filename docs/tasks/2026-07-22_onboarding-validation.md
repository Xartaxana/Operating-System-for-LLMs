# Валидация онбординга D-0090 — два полигона (DAG, D-0080)

Заказ: слово оператора 2026-07-22 «да, сделать валидацию на новом
проекте и на "живущем" D:\Dog». Процедура — по драфту
docs/tasks/2026-07-22_onboarding-architecture.md (чеканка D-0090).
Первая задача каждого полигона — evaluation, не build (D-0030).
Кит-артефакты (протокол, шаблон ledger) на момент старта НЕ
существуют (мораторий D-0074) — валидация гоняет процедуру
ВРУЧНУЮ по драфту; трение ручного прогона = прямой вход кит-батча.

Критерии успеха валидации (предрегистрация, до прогонов):
- V1: инвентаризация шага 0 заполняется по драфт-опроснику без
  до-изобретения вопросов (пробелы опросника = находки).
- V2: каждый кит-механизм получает статус в ledger без остатка
  «не знаю куда отнести» (остаток = дефект таксономии статусов).
- V3 (brownfield): ни одна установка слоя не требует изменения
  существующих носителей/потоков хоста (нарушение = дефект
  принципа 4).
- V4 (greenfield): полный комплект встаёт связками без ручного
  до-тюнинга предпосылок (ручной шаг = пробел манифестов).
- V5: расхождение процедуры с реальностью любого рода — находка в
  FINDINGS (класс F-51/F-35), не молчаливый обход.
