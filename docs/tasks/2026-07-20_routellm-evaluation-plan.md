# Методика оценки RouteLLM (первая задача Router-воркстрима)

Статус: ПЛАН (написан 2026-07-20 по слову оператора, до открытия
гейта; исполнение — по гейт-отчёту с подписью). Владелец: Lead.
Рамка ROADMAP: первая задача открытого Router-гейта — ОЦЕНКА
существующего инструмента, не стройка (D-0030/D-0033). Внешний
контекст: docs/RELATED_WORK.md «RouteLLM» (прайоры 2024; свежий
сюрвей — шаг 0 ниже).

## Вопрос оценки (единственный; ПЕРЕФОРМУЛИРОВАН 2026-07-20 по
слову оператора)

Выигрывает ли (или хотя бы не проигрывает) конструкция С РОУТЕРОМ
на allocate/dispatch-шаге против конструкции с Lead-координатором —
по ДЕНЬГАМ и ВРЕМЕНИ на принятую единицу при не-худшем качестве?

Гипотеза оператора, поддержанная нашей телеметрией: самое дорогое
в конструкции — КООРДИНАЦИОННАЯ работа (чек 11: доля Lead-токенов
на спеку/передачу/приёмку/журнал; экзамены №5–№10: «переключение
доминирует», B-плечо экономэкзамена — самое дорогое; из этой же
эмпирики родился батчинг D-0081). Роутер снимает с пути
фронтир-токены координатора и латентность его оборотов — приз
не «лучшие решения о маршрутизации», а УДАЛЕНИЕ ДОРОГОГО
РЕШАТЕЛЯ при сохранении качества решений.

Первая версия этого плана мерила роутер против
«политики-как-роутера» только по качеству маршрутных решений,
молчаливо считая координацию бесплатной — ошибка baseline,
исправлена. Прежний вопрос (маржинальная точность маршрутизации)
остаётся ВТОРИЧНОЙ метрикой шага 2.

Не-цели: интеграция в прод до вердикта; обучение своего роутера;
оценка на чужих бенчмарках (прайоры в RELATED_WORK); упразднение
приёмки (супервизия качества остаётся — снимается allocate/dispatch,
не контроль).

## Дизайн эксперимента: D-плечо экономического экзамена

Машинерия сравнения конструкций УЖЕ существует —
PROCESS/DEPLOYMENT_ECONOMY_EXAM.md (плечи A/B/C, exam_runner,
грейдинг-планки). Оценка роутера = НОВОЕ ПЛЕЧО D на той же батарее
задач: роутер (bert, оффлайн) назначает ярус каждой задаче
автоматически; воркер яруса исполняет headless; приёмка —
калиброванным судьёй по тем же планкам грейдинга (Lead не в петле
allocate/dispatch; качество-гейт сохраняется судьёй + выборочным
chief-judge аудитом).

Метрики на плечо (те же, что у A/B/C, для сопоставимости):
- $ на принятую единицу (полная, с петлями ретраев — Update Rule 4);
- wall-clock время до принятой единицы;
- качество по планкам грейдинга (та же шкала);
- ДОБАВКА D-плеча: доля координационных токенов (должна упасть с
  измеренной чеком 11 величины до ~цены инференса роутера + судьи).

Шаг 0-данные (Measure Before Optimizing, ДО прогонов): вытащить из
экзаменов №5–№10, чека 11 калибровок и savings_report ИЗМЕРЕННУЮ
долю координационных накладных — размер приза; если координация
< порога окупаемости (см. предрегистрацию), D-плечо не гоняется.

## Шаг 0 — Сюрвей (D-0066, двухпроходный)

scout/web: живость lm-sys/routellm на 2026 (или живой наследник
класса), лицензия, предобученные роутеры (mf/BERT/causal/SWR),
формат данных инференса и калибровки порога, оффлайновость
(КРИТИЧНО: требует ли mf-роутер embedding-вызовов внешнего API — это
меняет стоимость и приватность), OpenAI-совместимый серверный режим.
Затем прицельный второй проход Lead по перспективным местам; трейл —
в RELATED_WORK той же секцией. Гейт шага: проект мёртв И наследника
нет → вердикт «кандидата нет», оценка останавливается отчётом.

## Шаг 1 — Смоук локальной применимости (builder, ~$0)

Установка в venv; предобученный роутер на N>=50 РЕАЛЬНЫХ промптов из
requests.db + корпусов, sweep порога. Deliverable: таблица (промпт ->
score -> решение strong/weak при 3-5 порогах).

ПОПРАВКА ПО СЮРВЕЮ 0 (2026-07-20, второй проход Lead): кандидат шага
— `bert` (0.3B, оффлайн через transformers), НЕ рекомендуемый
upstream'ом `mf`: mf и sw_ranking требуют OPENAI_API_KEY для
эмбеддингов на инференсе (issue #62, встроенной альтернативы нет), а
OpenAI-ключа в нашем стеке нет и заводить его ради оценки — лишняя
зависимость. `causal_llm` (8B) — запасной, если bert не
дискриминирует (тяжёл для локальной GTX 1060, но CPU-прогон N=50
терпим). Риск-фактор в вердикт: upstream полуживой (коммиты 2024,
веса HF 2025-04) — adoption означает владение upkeep'ом.

## Шаг 2 — Оффлайн-оценка качества маршрутизации (builder + Lead)

Ground truth пары: вердикт наших судимых прогонов — equivalent =
«дешёвого хватило бы», worse/capability-reject = «нужен сильный».
Метрики:
- recall на «нужен-сильный» (ложный даун-рут = потеря качества);
- доля маршрутизации вниз (потенциал экономии);
- кривая порога: % вниз против misroute-rate.
Сравнение с тремя baseline: (а) всё-сильному; (б)
политика-как-роутер (категорийная статика D-0085-привязок); (в)
оракул по вердиктам (потолок).

ПРЕДРЕГИСТРАЦИЯ ключей (до прогона, дисциплина оси 8).
Вторичная метрика (этот шаг): паритет маршрутных решений — роутер
не хуже категорийной политики по misroute-rate (<=5% на
«нужен-сильный»).
ГЛАВНЫЕ ключи (D-плечо экономэкзамена, числа фиксируются после
шага 0-данных и ДО прогонов): D выигрывает, если $/принятую <=
B-плеча минус предрегистрированная дельта И wall-clock <= B, при
качестве по планкам не ниже B минус допуск; «не проигрывает» =
паритет по деньгам/времени при паритете качества (тоже валидный
позитивный вердикт — тогда решает простота конструкции). Если
шаг 0-данных покажет координационные накладные ниже порога
окупаемости — D-плечо не гоняется, вердикт «приза нет» с числами.

## Шаг 3 — Экономика R4 (Lead)

Стоимость роутера: инференс/запрос (замер шага 1), upkeep
(калибровка порога при смене привязок — по прецеденту judge-жизни),
амортизация самой оценки. Экономия: маржинальная добавка шага 2 ×
дельты цен категорий × объёмы R2-окна, проекция на месяц. Вердикт:
adopt только при экономии >= 3× стоимости ПРОТИВ ПОЛИТИКИ.

## Шаг 4 — Вердикт и носители

Возможные вердикты: (а) adopt как опциональный компонент тулкита
(порт-очередь, D-0074); (б) reject для наших деплоев — политика
достаточна (запись в RELATED_WORK «оценено, отклонено, триггер
переоткрытия»); (в) данных мало — named trigger добора. Носители:
evidence-строки в SHADOW_EVALUATION_LOG (адаптированный формат),
решение — DECISIONS при adopt/reject, трейл сюрвея — RELATED_WORK,
строка в гейт-отчёт.

## Бюджет и ярусы

Сюрвей — scout-класс (web, ~$0); шаги 1-2 — builder-класс, локально,
API-цена ~$0 (инференс роутера не зовёт LLM; судьи не нужны —
вердикты уже есть); шаг 3-4 — Lead. Итого оценка ~2-4 часа работ,
деньги ~$0. Детектор исполнения: строка «RouteLLM evaluation» в
гейт-отчёте — калибровка, следующая за открытием гейта, проверяет
наличие вердикта (чек 0/16 класса RELATED_WORK-хвостов).
