# Findings

Реестр эмпирических находок из эксплуатации самой архитектуры
(dogfooding) — оба контура: субскрипционный (Claude Code) и API
(gateway / Shadow Evaluation). В отличие от docs/RELATED_WORK.md
(внешние приоры, которые наша телеметрия должна подтвердить или
опровергнуть), здесь фиксируются наблюдения, полученные НАШИМИ
сессиями. Материал для White Paper.

Каждая находка: наблюдение → как получено → следствие для архитектуры →
статус (нужно ли решение DECISIONS.md).

Правила реестра. Записи дублируют-с-указателем, а не заменяют
канонический источник (evidence-лог DELEGATION_TABLE.md, отчёты в
docs/task_reports/ остаются на месте — D-0038); при расхождении
канонический источник прав. Нумерация — по порядку записи в реестр, не
по хронологии наблюдения: F-2..F-8 внесены 2026-07-08 задним числом,
при том что наблюдались раньше F-1.

## F-1 — Дефолтный харнесс не инициирует делегирование сам

**Наблюдение.** Оболочка Claude Code по умолчанию НЕ поощряет спавн
субагентов без явного запроса оператора. Инструкция агента прямо
задаёт: «Do not spawn agents unless the user asks.» Предоставленный сам
себе, Lead-сессия выполняет делегируемую работу САМА — на самом дорогом
из доступных ярусов (frontier-модель), — вместо того чтобы отдать её
вниз.

**Как получено (2026-07-08).** Delegated Task 6 (builder-tier по нашей
же DELEGATION_TABLE.md) ушла в субагента Sonnet и была принята с первого
ревью — но ТОЛЬКО потому, что оператор явно сказал «запускай в
субагенте». Без этой команды Lead (Fable) с высокой вероятностью
выполнил бы задачу сам. Диспатч был ручным: в этом репозитории на тот
момент не было ни CLAUDE.md с политикой, ни агентов scout/builder/critic,
ни журнала — политика роутинга существовала только как пилот на
D:\AO3_tests. Сопоставимо с первой живой проверкой на AO3 (2026-07-07):
там делегирование сработало, потому что дисциплина была прописана в
CLAUDE.md проекта; событие в журнал при этом не попало, т.к. диспатч
шёл через /qa-loop, чей промпт предшествовал политике. Оба случая — об
одном: **дисциплина действует ровно там, где она физически прописана в
загруженном контексте сессии.**

**Следствие для архитектуры.** Экономия от делегирования на
субскрипционном контуре НЕ возникает по умолчанию — её нужно активно
включить. Политику маршрутизации обязано автозагружать в контекст Lead
каждой сессии (в Claude Code это CLAUDE.md проекта); одних определений
агентов в .claude/agents/ недостаточно, потому что они лишь ОПИСЫВАЮТ
ярусы, но не побуждают Lead к ним обращаться. Это усиливает «Escalation
is a Resource» (PROJECT_PHILOSOPHY.md) конкретным механизмом: без
явного, всегда-загруженного правила модель делегирует по нулям, а не
избыточно. Практический вывод для проекта: routing MVP надо
разворачивать per-project (CLAUDE.md + агенты + журнал), а не считать,
что «умная модель сама догадается делегировать». Развёрнуто в этом
репозитории 2026-07-08 (эталонный деплой) и на AO3_tests (пилот).

**Замечание для White Paper.** Это контрапункт к нарративу
«агент-фреймворки максимизируют число агентов» (ANTI_GOALS.md): дефолт
современного продакшн-харнесса — противоположный, консервативный
(не-делегирование). Наша задача — не сдержать взрывной спавн, а поднять
делегирование с нуля до обоснованного минимума, управляемого evidence.

**Статус.** ОФОРМЛЕНО решением D-0041 (2026-07-08, восстановленный
Lead на Fable с одобрения оператора): делегирование на субскрипционном
контуре — opt-in; «задеплоить routing MVP» = политика в автозагрузке +
агенты ярусов + журнал, всегда втроём. Сама находка была записана в
degraded-режиме на Opus 4.8, решение — после lead_restored, как и
требует D-0039. Смежное решение той же датой: D-0042 (operator-switch
как триггер деградации).

## F-2 — Провайдерский кэш уже поглощает почти всё повторение контекста

**Наблюдение (2026-07-07, Task 5).** 97.6% input-токенов
субскрипционного контура — cache reads; учётная экономия против
безкэшевой цены $7,117 при общих тратах $1,178 (all-time базлайн,
8747 ходов).

**Следствие.** Прямое локальное подтверждение порядка D-0036: сначала
cache-aware учёт и измерение net-of-cache, и только потом какая-либо
lossy-компрессия. C-гейт Phase 2 обязан считать «деньги на столе»
только по ОПЛАЧЕННОМУ некэшированному повторно отправленному входу.

**Источник (канонический).** docs/task_reports/task-5_usage-report.md;
сводка в CURRENT_CONTEXT.md «Claude Code Baseline». Статус: учтено в
D-0036/ROADMAP (C-критерии cache-aware); решения не требуется.

## F-3 — Базлайн — цензурированные данные; frontier жжёт быстрее на единицу работы

**Наблюдение (2026-07-07, поправка Архитектора к Task 5).** Микс трат
базлайна нельзя читать как «где деньги» — оператор нормировал
использование frontier-ярусов из-за лимитов (цензурированные данные).
Корректное чтение — скорость расхода на единицу: opus-4-8 $0.264/ход,
fable-5 $0.216 против sonnet-4-6 $0.114 и sonnet-5 $0.063 (2–4x).

**Следствие.** Метрика успеха роутинга — стоимость за ПРИНЯТУЮ единицу
работы по ярусу + доля эскалаций, а не доля frontier в тратах
(освобождение лимитов легитимно ПОДНИМЕТ долю Fable на архитектурной
работе). Еженедельный цикл смотрит тренд свежего окна, не all-time.

**Источник (канонический).** docs/task_reports/task-5_usage-report.md
(полный текст поправки); сводка в CURRENT_CONTEXT.md. Статус: принято
как методика Phase 1.5; решения не требуется.

## F-4 — Вложенная диспетчеризация ломается на практике

**Наблюдение (2026-07-04, репетиция на AO3_tests).** Попытка дать
субагентам запускать собственных субагентов сломалась в живой
репетиции оркестрации QA-конвейера. Деталей прогона сохранилось мало
(находка предшествует и этому реестру, и журналу маршрутизации —
канонической записью была одна строка в CLAUDE.md пилота AO3_tests,
вне этого репозитория; поэтому здесь полная запись, а не указатель).

**Следствие.** Один из трёх аргументов D-0037 (плоское делегирование):
помимо принципиальных — декомпозиция и есть работа сильнейшего яруса;
вложенность разрушает атрибуцию стоимости/evidence — есть и грубо
практический: в текущем харнесе это просто ненадёжно. «Decomposable» —
категория эскалации, не право воркера делиться.

**Статус.** Оформлено D-0037 (2026-07-07). Урок реестра: находки,
живущие строкой в чужом файле, теряют детали — фиксировать сюда сразу.

## F-5 — Дешёвая метрика сходства меряет многословность, а не качество

**Наблюдение (2026-07-03, первые прогоны Shadow Evaluation).** difflib
(посимвольное сходство) как судья: 2 из 5 вердиктов неверны —
развёрнутый корректный ответ получает sim≈0 против лаконичного
эталона. Низкий sim ≠ хуже; ручное ревью опрокинуло rejected по coding.

**Следствие.** Валидация делегирования требует семантического судьи
(LLM), а судья — калибровки против размеченного людьми/Lead'ом сета.
Дешёвая детерминированная метрика годится только как грубый фильтр
высоких значений.

**Источник (канонический).** DELEGATION_TABLE.md, каверза
«comparison method» + Shadow Evaluation Log;
docs/task_reports/2026-07-03_shadow-evaluation-and-llm-judge.md.
Статус: закрыто внедрением LLM-судьи (D-0031).

## F-6 — Два режима отказа LLM-судьи: контаминация выборки и галлюцинация бага

**Наблюдение (2026-07-03).** (a) Неудачная калибровка судьи записала
его промпты как обычный трафик lead-gemini; случайная выборка Shadow
Evaluation втянула 6 вложенных судейских промптов из 11 пар — четыре
строки evidence отозваны (RETRACTED). (b) Судья middle-groq,
трассируя КОРРЕКТНЫЙ код фибоначчи, галлюцинировал баг («возвращает
b» — возвращает a) устойчиво к промпт-тюнингу; заменён reasoning-моделью
(judge-groq, калибровка 11/11).

**Следствие.** Судья — сам supervised worker (D-0031): выборка
исключает judge/replay трафик по traffic_kind; статус-меняющие
вердикты проходят chief-judge ревью; замена модели судьи — только по
измеренной деградации согласия. Режим отказа «галлюцинация при
трассировке корректного кода» внесён в правила critic-агента.

**Источник (канонический).** DELEGATION_TABLE.md Shadow Evaluation Log
(RETRACTED-блок и ruling); docs/task_reports/2026-07-03_shadow-evaluation-and-llm-judge.md;
PROCESS/JUDGE_CALIBRATION_PROTOCOL.md. Статус: оформлено D-0031.

## F-7 — Молчаливый $0: упавший учёт стоимости неотличим от бесплатности

**Наблюдение (2026-07-04).** Все строки evidence до 2026-07-03 несли
cost_target=$0.0000 — клиентский артефакт: litellm completion_cost()
не знал гейтвейных алиасов, молча бросал исключение, оно глоталось.
Проксайд-учёт в requests.db был верен всё время.

**Следствие.** Rule #1 проверяем только при честной цепочке стоимости:
неизвестная цена — это явный WARNING/None, никогда не $0; «не
вычислимо» печатается как «not computable yet + чего не хватает», а не
фальшивый ноль. Паттерн зашит в usage_report.py (unknown model → None
+ warning) и в спеку Task 3 (правило 3).

**Источник (канонический).** DELEGATION_TABLE.md, каверза
«cost_target=$0.0000». Статус: закрыто фиксом shadow_eval.py; принцип
воспроизводится во всех новых инструментах.

## F-8 — Слепые зоны телеметрии: параллельная раскладка транскриптов и рост живых файлов

**Наблюдение (2026-07-07/08, Task 6).** (a) Субагентные транскрипты
живут в отдельной раскладке (<project>/<session>/subagents/*.jsonl) и
были целиком невидимы для cc_usage — 7.2% токенов / $100.03 учётных
не попадали в книги; ретро-базлайн AO3 занижен на $57.82. (b) Живой
транскрипт растёт между командами: наивная проверка идемпотентности
импорта «дважды подряд» ловит 1–4 «новых» строки от САМОЙ работающей
сессии — сравнение honest только внутри одного вызова (снапшот).

**Следствие.** Полноту телеметрии надо верифицировать против самого
измеряемого процесса (чем больше делегируешь — тем сильнее врали бы
книги: недоучёт рос вместе с успехом роутинга). Идемпотентность на
живой системе определяется относительно фиксированного снапшота.

**Источник (канонический).** docs/task_reports/task-6_subagent-transcripts.md.
Статус: закрыто фиксом Task 6 (коммит 75af5b5); решения не требуется.

## F-9 — Автозагруженная политика не самоисполняется: Lead молча самоизъялся в первые же часы

**Наблюдение (2026-07-08, поймано оператором).** Через несколько часов
после усиления правила 1 («разведка → scout по умолчанию») Lead-сессия
выполнила разведочную проверку (поиск по репозиторию + чтение файла)
сама, без scout — и БЕЗ требуемой правилом записи причины в журнал.
Пропуск по существу был оправдан (4 точечные команды по известным
целям; передача контекста scout'у дороже самой работы — Rule #1), но
он был молчаливым: нарушение заметил оператор, не система.

**Следствие.** F-1 говорил: без автозагруженной политики делегирование
не возникает. F-9 уточняет: автозагруженной политики тоже мало, если
соблюдение дороже нарушения. Правило «всегда делегируй разведку» без
нижней границы срабатывает на каждый grep — и потому игнорируется, а
игнорируемое правило не производит данных. Механизм исправления —
сделать ПРОПУСК дешёвым и легальным: явный тип события
`dispatch_skipped` (одна строка в журнал с причиной) + калибровка
правила (точечная сверка ≤~4 известных целей может остаться у Lead,
скрытая разведка неизвестного объёма — всегда scout). Молчаливый
пропуск остаётся нарушением; журналируемый — данными о реальной
границе яруса, которые читает еженедельная калибровка. Для White
Paper: комплаенс-действие должно быть дешевле нарушения, иначе
политика деградирует в текст; и надзор оператора ловит то, что
самонадзор пропускает (человек в контуре — не декорация).

**Источник (канонический).** logs/routing-log.jsonl (событие
dispatch_skipped 2026-07-08T13:24:36, RETROACTIVE); калибровка правила
1 и тип события — CLAUDE.md обоих деплоев. Статус: механизм внедрён и
в тот же день (по вопросу оператора) РАСПРОСТРАНЁН НА ВСЕ ЯРУСЫ —
универсальное правило 8 в CLAUDE.md обоих репо: любая задача,
отображающаяся на дешёвый ярус, но выполненная Lead'ом сам, законна
только с dispatch_skipped (льгота: critic-пропуск на мелком диффе —
пометкой внутри accepted-события; пропуски правил /qa-loop покрывает
orchestrator-log). Решения DECISIONS.md пока не требуется (если
dispatch_skipped приживётся, вокабуляр журнала стоит зафиксировать
решением при первой недельной калибровке).

## F-10 — Все ярусы и модели чинят экземпляр дефекта, а не класс

**Наблюдение (2026-07-08, оператор; паттерн ловился неоднократно,
кросс-модельно).** Найдя дыру в одном месте, модель чинит её там же —
и не обобщает: не называет класс дефекта, не проходит по известным
аналогичным местам, не создаёт общего правила. Три подтверждения за
одну эту сессию: (а) лекарство F-9 применено только к scout, хотя
builder/critic имели ту же дыру — обобщено лишь после вопроса
оператора; (б) routing MVP месил только AO3, собственный репозиторий
проекта стоял без политики, пока оператор не спросил «а мы сами?»;
(в) триггер D-0042 добавлен в один деплой и не синхронизирован во
второй до следующего касания. Каждый раз аналогичные места были
известны или обязаны были быть известны.

**Следствие.** Локальное латание — устойчивый режим отказа LLM,
воспроизводящийся на всех ярусах (от Haiku до frontier) и не
исправляемый ростом модели; значит, обобщение фикса должно быть
явной обязанностью системы, а не ожиданием от интеллекта. Оформлено
D-0043 «Fix the class, not the instance» на КАЖДОМ уровне, где живут
правила поведения: конституция (SYSTEM_PROMPT.md — три шага: назови
класс / обойди аналоги, чиня или явно ставя в очередь / правило — на
самый верхний связывающий уровень), политика (правило 9 в CLAUDE.md
обоих деплоев), роли (builder докладывает замеченные аналоги, не
расширяя scope — D-0037; critic проверяет классовую полноту фикса как
штатную категорию находок; Lead владеет обходом). Молчаливо
оставленный известный аналог = нарушение — симметрично молчаливому
пропуску F-9. Для White Paper: пара F-9/F-10 — два лика одной
проблемы (локальность поведения LLM); лечение одно — сделать
глобальное действие дешёвым, явным и проверяемым ревьюером.

**Источник (канонический).** DECISIONS.md D-0043; SYSTEM_PROMPT.md;
CLAUDE.md и .claude/agents/{builder,critic}.md обоих деплоев. Статус:
оформлено D-0043 (2026-07-08). Остаток обхода поставлен в очередь
честно: девять QA-агентов конвейера AO3 получат строку «докладывай
замеченные аналоги» при следующем касании их промптов (записано в
CURRENT_CONTEXT.md).
