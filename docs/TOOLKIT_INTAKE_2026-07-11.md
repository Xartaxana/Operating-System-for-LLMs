# Toolkit: интейк новой цели (направление оператора, 2026-07-11)

Статус: ИНТЕЙК — формализация направления + карта готового/недостающего
+ открытые вопросы оператору. Не план строительства: первая задача
воркстрима — оценка существующего (D-0030/D-0033), она уже идёт
(разведка t-044: ядро vs догфудинг vs смешанное). Порядок стадий —
D-0059 (intake → scope → декомпозиция → allocate), артефакты раньше
кода.

## Цель (словами оператора, зафиксировано)

Система становится ИНСТРУМЕНТОМ для любых других пользователей:

1. Доделать технические хвосты.
2. Завернуть систему (возможно — новый репозиторий) так, чтобы она
   была инструментом разработки ДРУГИХ проектов с нуля или в
   действующем проекте — а не самой себя. Пользователь не видит наших
   задач и планов расширения; их мы ведём сами у себя.
3. Лёгкая установка: с нуля или в существующий проект.
4. Лёгкий способ назначить модели на роли/ярусы; модели
   ЭКЗАМЕНУЮТСЯ; дальше делегирование «работает само», пользователь
   пишет свой проект.
5. Уведомления пользователя отнимают минимум внимания и приходят
   только в нужный момент.

## Что уже есть (карта на сегодня)

- **«Политика и есть роутер»** (ARCHITECTURE Portability, D-0005/
  D-0062): порт системы = порт ПОЛИТИКИ, не софта. Деплой-тройка
  уже определена (D-0041): автозагружаемая политика + role-агенты +
  журнал. Двойное доказательство переносимости: AO3-пилот (второй
  деплой, живёт неделю) и само это репо.
- **Назначение моделей на роли** — ровно D-0062 (функция→модель —
  свойство деплоя; мост «Two Vocabularies»). Пользовательский шаг
  «выбери модели» = заполнение привязки; словарь правил не меняется.
- **Экзамены при назначении** — активы существуют: входной фильтр
  scout (golden set, D-0057), ранжирующий Lead-экзамен
  (PROCESS/LEAD_RANKING_EXAM.md), калибровка судьи
  (JUDGE_CALIBRATION_PROTOCOL, D-0031), детектор фабрикации
  pi_run_guard (t-017) для не-Claude воркеров.
- **«Работает само»** — политика CLAUDE.md (правила 1–11) +
  код-гейты (валидатор журнала, mechanism-гейт, квотные стены,
  preflight) + SessionStart-хук «реальность фоном».
- **Минимум внимания** — уже принципы системы: фоновые диспатчи
  (D-0040), Boot Report как единая точка утреннего статуса, WARN/
  BREACH-пороги. Осталось оформить как продуктовый контракт (см.
  разрывы).
- **Два контура**: Claude Code (subscription) — зрелый; API-контур
  (gateway) — лаборатория, но это и есть вариант для пользователей
  без Claude Code.

## Известные разрывы (до t-044; после разведки список уточнится)

- **Р1. Сцепленность ядра с догфудингом**: наша политика/скиллы/боты
  ссылаются на CURRENT_CONTEXT/ROADMAP/FINDINGS — пользовательская
  версия должна жить без них (карту сцепок даст t-044 Q2).
- **Р2. Repo-специфичность экзаменов**: вопросы scout-набора привязаны
  к НАШИМ файлам; для чужого репо нужен либо генератор набора при
  установке (Lead-модель пользователя генерирует и пинит ключи —
  протокол D-0057 это уже описывает как процедуру), либо
  универсальный вступительный минимум + генерация локального набора
  после (t-044 Q3 уточнит объём).
- **Р3. Инсталляция**: сегодня деплой = ручной порт Lead-сессией
  (AO3-прецедент). Нужен упаковка+установщик (см. варианты ниже).
- **Р4. Онбординг привязки моделей**: файла-конфига «роль→модель»
  как единой точки нет (в Claude Code это frontmatter агентов; в
  gateway — config.yaml алиасы) — нужен один пользовательский шаг.
- **Р5. Продуктовый контракт уведомлений**: когда системе МОЖНО
  дёргать пользователя (приёмка не проходит дважды, эскалация выше
  верхнего яруса, бюджет BREACH, экзамен провален) — списком, всё
  остальное фоном/в отчёт.
- **Р6. Технические хвосты очереди**, попадающие в ядро: A3
  (манифест контекста диспатча), structured worker-frames (Rule #1 —
  для продукта станет актуальным), critic golden set, AO3-порт
  by/basis (ось 1), admission-математика стены (evidence-gated).
  Хвосты, НЕ входящие в ядро, остаются нашим догфудингом.

## Варианты упаковки (решение — оператора)

- **B1. Новый репозиторий-шаблон (template repo)**: чистое ядро
  (политика, agents/, skills/, tools/, .githooks/, PROCESS-ядро,
  экзамены) + install-скрипт «в существующий проект» (копирует
  тройку деплоя, инициализирует журнал, ставит hooksPath, запускает
  привязку и экзамены). Наш репо остаётся штабом разработки ядра;
  ядро синхронизируется в шаблон релизами. ПЛЮС: полная изоляция
  наших планов (требование 2 буквально); МИНУС: двухрепозиторная
  синхронизация (ось в SIBLING_MAP + чек живости).
- **B2. Каталог dist/ внутри этого репо** + скрипт сборки шаблона:
  один источник, пользователь берёт артефакт релиза. ПЛЮС: нет
  второго живого репо; МИНУС: пользователь видит наш репозиторий
  (нарушает требование 2, если не публиковать только артефакт).
- **B3. Claude Code plugin** (marketplace: skills+agents+hooks одним
  пакетом): самый нативный способ установки для Claude Code-контура.
  МИНУС: не покрывает API-контур; формат диктует харнесс.
- Рекомендация Lead: **B1 (+B3 позже как канал доставки того же
  ядра)** — единственный вариант, буквально выполняющий «пользователь
  не видит наших задач», и он же дешевле всего валидируется третьим
  деплоем (ZCRT-аналог: чистая установка на пустой проект).

## Порядок воркстрима (по D-0033: сначала оценка, потом стройка)

1. t-044 инвентаризация (идёт) → карта ядро/догфудинг/сцепки.
2. Решения оператора по открытым вопросам (ниже).
3. Спека ядра v0: состав файлов шаблона + install-путь + онбординг
   привязки/экзаменов + контракт уведомлений (Lead, по карте t-044).
4. Скелет шаблона (builder по спеке) + установка на ПУСТОЙ
   тест-проект (третий деплой = валидация, ZCRT-аналог).
5. Установка в ДЕЙСТВУЮЩИЙ проект (четвёртый деплой-тест).
6. Только после обоих прогонов — публичная упаковка (README
   пользователя, лицензия, релиз).
