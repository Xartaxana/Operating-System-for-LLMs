# kit-release — релиз кита (D-0097)

Единственный владелец релизной процедуры. Кодифицирует уроки релизов
v0.1.0 (снимок+тег), v0.4.0–v0.4.2 (release-гейт critic, паритет
правил), v0.5.0 (гейт калибровкой, инвентаризация t-294, пре-релизный
батч t-295, моды хуков клона). Порядок жёсткий: гейт → инвентаризация
→ мелочи → сборка → пост-обязанности. Пропуск шага — не ускорение,
а класс уже случившейся ошибки (каждый шаг ниже несёт свой прецедент).

## 0. Предусловия (проверить, не предполагать — F-30)

- Слово оператора на релиз получено (вызов скилла = это слово).
- РЕЛИЗНЫЙ ГЕЙТ пройден и назван: какой именно (critic release-gate
  как v0.4.0/t-214, или калибровочный аудит как v0.5.0 — чеки
  30/31/экономика). Гейт не пройден → СТОП, доложить оператору.
- Ярус сессии — Lead (D-0056а): релиз двигает механизмы и публичный
  репозиторий.
- Чек 14(е): большой экзамен экономии деплоя — «периодически, не
  чаще раза на релиз». Если БОЛЬШОГО прогона нет уже несколько
  релизов подряд — вопрос оператору ДО сборки, не молча.

## 1. Инвентаризация staging против порт-очереди (урок t-294)

Scout-диспатч (лист; приёмка по D-0094): по КАЖДОМУ пункту
накопленной порт-очереди (CURRENT_CONTEXT: блоки «порт-очередь»,
«батч мелочей», кросс-находки) — вердикт «уже в staging (file:line)»
/ «отсутствует», негативы только с позитивным одноформенным
контролем. Урок Dog: reconcile, don't apply twice — повторный порт
уже стоящего в staging ломает твины. Выход: два списка — «в работу»
(мелочи этого релиза) и «named-узлы следующего минора» (крупное, не
блокирующее релиз, — остаётся очередью с именами).

## 2. Пре-релизный батч мелочей

- Кодовые твины — ОДНИМ builder-батчем (D-0081, маркер «батч
  мелочей»); DoD по чек-листу D-0096 (свежесть вводных — против
  носителя, не памяти: прецедент П5 t-289, спека на протухшей
  заметке).
- Политика кита = ядро штаба: правила портируются ПО СМЫСЛУ, без
  штабных не-резолвящихся якорей (t-NNN, номера штабных чеков) —
  переименовывать дескриптивно; словарь кита английский. Проверка
  якорей — явный вопрос критику (урок t-295).
- Critic-гейт по R3 на весь дифф батча (код + политические порты).
- Витнесс подтверждается на ФИНАЛЬНОМ дереве контрольным перегоном
  Lead (обе обвязки: канон штаба + toolkit/tools/), НЕ только словом
  builder'а: урок WITNESS ECHO v0.5.0 — красный трек позже витнесса
  разрешается только перегоном.

## 3. Сборка снимка

1. Клон публичного репо: D:\Improving_AI\Supervised-Delegation;
   `git status` чист, ветка main на origin. Грязный клон — разобрать
   ДО копирования.
2. Копирование БЕЗ зеркальной деструкции (авто-классификатор
  блокирует robocopy /MIR — и правильно): `cp -r toolkit/. <клон>/`
   + чистка `__pycache__`/`.pytest_cache` + `git add -A`.
3. УДАЛЕНИЯ сверяются явно: `git status --short | grep '^D '` —
   каждый D осмыслен (файл реально ушёл из staging), не артефакт
   копирования.
4. EXEC-БИТЫ ХУКОВ В ИНДЕКСЕ КЛОНА (урок v0.5.0, D-0093): копия ФС
   на Windows моду НЕ переносит — `git update-index --chmod=+x`
   на оба .githooks-файла; контроль `git ls-files -s .githooks` =
   100755 оба. Релиз со 100644 = молча мёртвые гейты у
   Linux-потребителей (класс F-53).
5. Release-коммит: публичный стиль (EN, «vX.Y.Z: <дельта с прошлого
   тега>» — смотри git log клона), в теле — группы дельты; строка
   «Staff snapshot: <хэш штабного коммита>». Гейты клона могут
   потребовать осевой блок — тогда skip-строкой кита «axes: not a
   mechanism (release snapshot of shipped files)» (прецедент
   INSTALL.md).
6. Аннотированный тег vX.Y.Z + `git push origin main --follow-tags`;
   контроль: log -1 и тег на origin.

## 4. Пост-релизные обязанности (тем же ходом, D-0082)

- СНИМОК-РЕВИЗИЯ для D-0091-леджеров хостов = vX.Y.Z/<публичный
  хэш> — записать в CURRENT_CONTEXT; хосты с леджером без ревизии
  (доверсионные установки) берут её при первом sync — их INBOUND
  обязан это предписывать.
- CURRENT_CONTEXT: блок «релиз отложен/за гейтом» → «вышел» (все
  упоминания, не только основной блок); named-узлы следующего минора
  — явной очередью; мораторий D-0074 продолжает действовать.
- Журнал: диспатчи батча закрыты приёмками; сборка снимка —
  детерминированная операция среды (D-0095, skip-событий не нужно),
  след — коммиты обоих репо.
- Кросс-пункты деплоям (если релиз несёт для них порт-строки) — в
  ИХ носители тем же ходом (D-0082).

## Детекторы (D-0049)

Шаговые отказы ловятся внешними детекторами, зарегистрированными до
этого скилла: моды хуков цели — чек 32(г) (D-0093); релиз без
экзаменной строки — чек 14(е); дрейф твинов штаб↔кит — чек 22
кит-протокола и ось 7 карты; пропуск самого скилла при релизе —
обратный проход чека 24 (нарратив релиза в носителях против
процедуры-владельца) и рецидив любого из кодифицированных классов.
