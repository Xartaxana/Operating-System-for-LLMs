# /permission-audit — разбор permission-запросов

Порт из деплоя AO3 (оценка t-106: пилот на живой сессии поймал
позитивный контроль и вскрыл доминирующее нарушение), принят t-107.
Скрипт: `tools/permission_audit.py` (+ снапшот транскриптов до скана,
+ MASKED-детектор широких правил). Пользователь НЕ должен присылать
скриншоты диалогов: сами диалоги в транскрипты не пишутся, но каждый
tool_use записан, а правила (allowlist + sandbox-эвристики) известны —
значит можно вычислить, какие вызовы требовали подтверждения.

**У аудита ДВЕ равнозначные цели, не одна:**
1. **Убрать шум** — чтобы законные повторяющиеся команды не спрашивали каждый раз.
2. **Вскрыть неправильные действия** — permission-запрос это не только помеха, это
   ДЕТЕКТОР. Агент (или ты сам) сделал что-то нештатно, и именно поэтому команда не
   совпала с allowlist. Чаще всего за запросом стоит не «забыли разрешить», а
   «сделали не так»: изобрели свою форму вместо готовой функции/канона, полезли
   править файл в обход Edit/Write, пропустили обязательное поле журнала,
   запустили ручной poll-цикл там, где для этого есть проверка. НЕ проскакивай
   мимо этого к «добавить паттерн» — сначала спроси по каждой группе: *почему эта
   команда вообще возникла в такой форме, и правильно ли это было?* Разрешать
   (глушить) можно только то, что было СДЕЛАНО ПРАВИЛЬНО. Неправильное — чинить у
   источника (функция/скрипт/инструкция агента), а не легитимизировать в allowlist.

## Шаги

1. Запусти аудит: `python tools/permission_audit.py --minutes 180`
   (окно подбери по ситуации; `--all` — вся история; `--summary` — сводка;
   `--session <id>` — фильтр по сессии). Скрипт сканирует транскрипты основной
   сессии И всех субагентов, прогоняет команды через allowlist
   `.claude/settings.json`/`settings.local.json`, встроенные auto-allow и
   sandbox-эвристики, и печатает подозрительные вызовы с причиной. Первым
   блоком печатается **MASKED-BY-BROAD-ALLOWLIST** (п.3) — прочитай его до
   основного списка: широкие правила гасят часть находок ещё до того, как ты
   их увидишь.

   **Снапшот-оговорка.** Скрипт фиксирует список транскриптов и их размер ДО
   скана и читает только эти байты — команды, дописанные после старта скрипта
   (в т.ч. самим прогоном аудита или параллельной сессией), в вывод не
   попадут; это осознанный компромисс (числа не плывут внутри прогона).

2. **Диагностический проход — СНАЧАЛА ищи неправильные действия, потом глуши шум.**
   Для каждого вызова: «глушить (сделано верно, просто не разрешено)» или
   «чинить у источника (сделано не так)». Сигналы дефекта — таблица под наши
   формы (CLAUDE.md, командная гигиена):

   | Сигнал в транскрипте | Вероятное неправильное действие | Фикс у источника |
   |---|---|---|
   | `python - <<EOF`, `python -c "...replace/open(...,'w')..."` по трекаемому файлу | правка файла в обход Edit/Write (гигиена п.4) | переделать через Edit/Write; напомнить правило воркеру |
   | `printf`/`echo >>`/`cat <<EOF` в `logs/routing-log.jsonl` | ручная сборка строки журнала вместо Edit/Write-тула (гигиена п.5); риск пропуска обязательных полей | запись строго Edit/Write-тулом; проверить поля уже записанных строк |
   | ad-hoc `grep`/`cat`/`python -c` для чтения/поиска | Bash вместо специализированного тула | Read/Grep/Glob вместо Bash |
   | `cd X && ` спереди или ` 2>&1` сзади у многих вызовов | форма ломает совпадение с allowlist (гигиена п.3) | не префиксовать cwd, не тащить `2>&1` |
   | разноформенный `python -m pytest ...` у каждого прогона | канон не используется | канон `python -m pytest tools/ gateway/ -q` из корня (п.1); узкий таргет — только оправданный задачей |
   | ручной poll-цикл (`while`/`for ... curl/sleep`) | своя форма ожидания вместо готовой | именованный скрипт `tools/` при повторении |
   | `git push`/установка пакетов/правка вне `owns` манифеста, которых не просили | действие вне мандата — запрос сработал как страховка | НЕ разрешать; разобраться (выход за манифест D-0073) |

   Всё из правой части — находка, отдельным списком: это и есть «что делается
   неправильно». Только ОСТАВШЕЕСЯ идёт в глушилку шага 4.

3. **Блок MASKED-BY-BROAD-ALLOWLIST — не про шум, про слепую зону.** Правила
   вида `Bash(python *)`, `Bash(python -c ' *)`, `Bash(python -m *)`,
   `Bash(bash -c *)` (в т.ч. с env-префиксом `VAR=1 ...`) пропускают
   ПРОИЗВОЛЬНОЕ выполнение: под них подходит что угодно из правой колонки
   шага 2 — и проходит МОЛЧА, не попадая в suspects вовсе. Прецедент
   (t-106): `settings.local.json` этого репо нёс сразу три таких записи.
   Такое правило — самостоятельная находка уровня «неправильное действие»:
   сузить или удалить (сужение = больше подтверждений — решение оператора,
   это его накопленные Allow-always).

4. Оставшееся — по причине, РАЗНЫЕ фиксы:
   - **«нет совпадения с allowlist»**, узкая повторяющаяся команда → wildcard в
     `permissions.allow` (`*` — префикс, совпадение С НАЧАЛА строки; cd-префикс
     ломает). НИКОГДА — паттерны произвольного выполнения (п.3).
   - **«многострочная / цикл / $(...)»** → allowlist не поможет (sandbox
     «cannot be statically analyzed»); повторяющуюся логику — в именованный
     скрипт `tools/`.
   - **Собственные ad-hoc чтения координатора** → Read/Grep/Glob; разовые
     команды в allowlist не тащить.

5. **Накопление `.claude/settings.local.json`.** В этом деплое ВЕСЬ
   `permissions.allow` живёт здесь (версионируемого костяка нет) — разбор
   равен разбору всего списка: дубликаты широких правил — удалить; повторы без
   обобщения — один wildcard; разовая история расследований — удалить; записи,
   легитимизирующие антипаттерн правой колонки — удалить (антипаттерн в allow
   выполняется молча — то же ослепление, что MASKED, только локальное). Итог —
   короткий список высокоуровневых правил. JSON-валидность после правки
   (`python -m json.tool`).

6. Сводка ДВУМЯ блоками (не сливать): **Неправильные действия (главное)** —
   что, у скольких вызовов, что починено у источника; «ничего не найдено» —
   тоже результат, сказать явно. **Шум** — сколько, топ-причины, что
   добавлено/сужено/удалено, записей было/стало.

## Границы
- Не ослаблять безопасность: никаких паттернов произвольного выполнения,
  `git push`, установки пакетов, правок вне манифеста owns.
- Аудит эвристический: «Allow once» и авто-разрешённые неразличимы — вывод
  «вероятно требовало подтверждения», не точный лог.
- В worktree-сессии аудит смотрит транскрипты КАНОНИЧЕСКОГО деплоя
  (PROJECT_KEY фиксирован), не самого worktree (граница F3 ревью t-107).

## Смежный класс, который аудит НЕ ловит
Дефект вызова без permission-запроса: вывод об ОТСУТСТВИИ объекта из
пустого/ошибочного вывода неправильно вызванного тула (`command not found`,
пустой grep своей формой). Это класс «env-негатив требует сверки» (гигиена
п.6, F-30/F-34): встретив в транскрипте негативное утверждение без позитивной
сверки рядом — фиксируй как находку правой колонки, даже без запроса.
Основной детектор класса — приёмка и калибровка; этот аудит — вторичная сеть.
